# 레거시 구조상 주요 리스크와 해석 메모

## 1. 목적

본 문서는 현재 `linkagelab-a11y-workmanage` 구조를 2차 개발 기준으로 읽을 때 주의해야 할 기술적/운영적 리스크를 정리합니다.

## 2. 핵심 리스크 요약

1. 비밀정보와 DB 접속 정보가 소스 코드에 하드코딩되어 있습니다.
2. SQL 문자열 결합이 광범위해 입력값 기반 취약점 위험이 큽니다.
3. 권한 모델이 화면/라우트/API에서 완전히 일치하지 않습니다.
4. 현재 범위에서 제외된 기능(아지트 알리미, 프로젝트)이 코드에 남아 있어 해석 혼선을 줍니다.
5. 리소스/검색/validation이 모두 `TASK_TBL` 문자열 데이터 품질에 직접 의존합니다.

## 3. 코드상 구체 이슈 메모

### 3.1 JWT/DB 비밀정보 하드코딩

- `server/config/config.js`에 `jwtSecret`, DB host/user/password가 직접 들어 있습니다.

영향:

- 코드 유출 시 즉시 보안사고로 이어질 수 있습니다.
- 2차 개발에서는 비밀정보를 환경변수/시크릿 매니저로 분리해야 합니다.

### 3.2 SQL 문자열 결합 다수

- `api_auth_controller.js`, `api_member_controller.js`, `api_a11y_work_controller.js`, `api_a11y_agit_controller.js`에서 문자열 템플릿으로 SQL을 직접 생성합니다.
- 일부 insert/update만 placeholder를 쓰고, 대부분 검색/삭제/validation은 raw string입니다.

영향:

- SQL injection 위험
- 입력 escape 누락 시 장애 가능
- 컬럼/테이블 변경 영향도 급증

### 3.3 로컬 토큰 만료 판정 버그

- `front/src/utils/auth.js`의 `isTokenExpired()`는 토큰에 `exp`가 있을 때 `null`을 반환하는 구조입니다.

영향:

- 로컬에서 토큰 만료를 제대로 걸러내지 못할 수 있습니다.
- 실제 유효성은 서버 `auth/check`에 사실상 의존합니다.

### 3.4 Axios 에러 처리 버그 가능성

- `front/src/apis/index.js`의 response interceptor는 `Storage.clearToken()`을 호출하는데, 해당 객체/함수 정의가 없습니다.
- `originalRequest.validateStatus()` 호출도 일반적인 axios 사용 방식과 다릅니다.

영향:

- 인증 오류 처리 중 2차 오류가 날 수 있습니다.
- 로그인 만료 UX가 불안정할 수 있습니다.

### 3.5 권한 경계 불일치

- 상단 메뉴에서 `보고`는 로그인 사용자에게 항상 보입니다.
- 하지만 `Report.vue`는 `memberLv > 0`일 때만 실제 내용을 렌더링합니다.
- 서버는 개인 업무 수정에서 `userLevel === 0` 사용자도 본인 데이터 수정은 허용합니다.

영향:

- 화면 권한, 메뉴 권한, 서버 권한이 서로 다릅니다.
- 2차 개발에서는 역할 모델을 먼저 명세해야 합니다.

### 3.6 관리자 화면의 즉시 파괴적 조작

- 삭제/비밀번호 초기화/일괄 치환 대부분이 `confirm()` 후 즉시 실행됩니다.
- 별도 감사로그, soft delete, 재검토 단계가 없습니다.

영향:

- 운영 실수 복구가 어렵습니다.
- validation bulk update는 특히 되돌리기 어렵습니다.

### 3.7 `FlashMsg` 구현 결함 가능성

- `front/src/components/FlashMsg.vue`는 `_`를 직접 사용하지만 import가 없습니다.

영향:

- 런타임 환경에 따라 토스트 제거 로직이 실패할 수 있습니다.

### 3.8 `app.js` 포트 계산 우선순위 주의

- `server/app.js`의 포트 계산식은 삼항 연산자 우선순위에 의존합니다.

영향:

- `PORT` 환경변수 주입 시 기대와 다르게 동작할 여지가 있습니다.
- dev/prod 포트 정책을 명시적으로 다시 쓰는 편이 안전합니다.

### 3.9 Webpack 설정 품질 저하

- `server/webpack.config.js`의 `test`, `exclude`가 정규식 literal이 아니라 문자열입니다.

영향:

- 빌드 동작이 운 좋게 넘어가는 수준일 수 있습니다.
- 2차 개발에서는 번들러 설정을 새로 정리하는 편이 낫습니다.

### 3.10 접근성/대용량 데이터 UX 상충

- `ReportSearch.vue`, `Search.vue`에는 기간이 길면 스크린리더가 멈출 수 있다는 주석성 문구가 들어 있습니다.
- 즉, 현재 구조는 대량 표 렌더링과 접근성 요구가 충돌하고 있습니다.

영향:

- 2차 개발에서 가상 스크롤, 페이지네이션, 요약 우선 전략 검토 필요

### 3.11 프로젝트 영역의 반쯤 남은 상태

- `Project.vue`와 `components/project/*`가 존재하지만 라우터에서는 비활성입니다.
- 프로젝트 목록/삭제/생성 코드는 남아 있으나 수정/정합성 보장까지 완성된 구조로 보기 어렵습니다.

영향:

- 2차 개발 범위에 넣을지 말지 먼저 결정하지 않으면 문서와 구현이 충돌합니다.

### 3.12 아지트 QA알리미 잔존 코드

- `AdminAgitNoti.vue`, 관련 API, `AGITNOTI_*` 테이블 인터페이스가 남아 있습니다.
- 사용자 판단 기준으로는 현재 스펙아웃입니다.

영향:

- 코드가 존재한다는 이유만으로 전환 범위에 넣으면 안 됩니다.
- 제거 후보 또는 완전 분리 후보로 따로 다뤄야 합니다.

### 3.13 Type / Service Group 관리모드 잔존 코드

- `AdminWorkTypeManage.vue`, `AdminSvcManage.vue`와 해당 CRUD API가 그대로 남아 있습니다.
- 하지만 현재 판단 기준으로는 두 관리모드 모두 스펙아웃입니다.

영향:

- `TYPE_TBL`, `SVC_GROUP_TBL`은 여전히 validation과 입력 보조에 필요하지만, 관리 UI CRUD까지 자동 복원하면 범위가 커집니다.
- 2차 개발에서는 `read/validation`과 `master CRUD`를 명확히 분리해서 범위를 고정해야 합니다.



## 4. 유지보수 시 해석 원칙

1. `TASK_TBL` 값이 더럽다는 전제를 두고 화면/집계를 읽어야 합니다.
2. 관리자 기능은 단순 CRUD보다 `정합성 보정` 성격이 강합니다.
3. 프로젝트와 아지트 알리미는 "코드 존재"와 "현행 범위 포함"을 구분해야 합니다.
4. DB 설계는 기존 PHP 문서를 베이스로 보되, 실제 사용 축은 이 저장소 기준으로 다시 좁혀야 합니다.

## 5. 이해를 돕는 결론

이 저장소의 복잡도는 Vue/Node 분리 자체보다 다음 네 축에서 나옵니다.

- `TASK_TBL`을 중심으로 한 입력/검색/집계/보정의 중첩
- `TYPE_TBL`, `SVC_GROUP_TBL`이 업무 데이터 품질과 직접 연결되는 구조
- 권한/UX/서버 검증 불일치
- 비활성/스펙아웃 기능이 코드에 잔존한 상태

따라서 2차 개발은 화면 복제보다 `범위 확정 -> 데이터 계약 재정의 -> 관리자 보정 기능 재설계` 순서로 접근하는 것이 맞습니다.

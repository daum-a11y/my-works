# 마이그레이션 원복 작업 계획서

작성일: 2026-03-27  
작성 위치: `my-works`  
비교 기준:

- 현재 이관 코드: `/Volumes/workspace/workspace/my-works`
- 원본 코드(사용자 PHP): `/Volumes/workspace/workspace/link/php-operation_tool`

## 1. 목적

이번 작업의 목표는 `기능 개선`이나 `신규 설계`가 아니라, 원본 기준 기능 복원이다.

- 원본에 있던 기능은 현재 이관 코드에서도 동일하게 동작해야 한다.
- 원본에 없던 기능은 제거 대상이다.
- 인증 체계 변경으로 인한 구조적 차이는 현재 이관 코드 구조를 유지한다.
- 화면 배치와 구조는 원본 기준으로 최대한 동일하게 맞춘다.
- 버그 수정은 필요한 범위만 최소 대응한다.

## 2. 이번 작업에서 고정할 원칙

### 유지할 것

- React/Supabase 기반 인증 구조
- 현재 저장소의 라우팅/앱 셸 뼈대
- 인증 변경 때문에 불가피하게 달라진 데이터 접근 방식

### 원복할 것

- 페이지별 화면 배치
- 입력/조회/수정/삭제 흐름
- 원본에 존재하는 필드, 버튼, 표 컬럼, 경고 문구, 빠른 이동, 다운로드, 정렬, 인라인 편집
- 원본 기준 권한별 노출 차이

### 하지 않을 것

- 원본에 없는 신규 기능 추가
- 단일 페이지 복구 작업 중 주변 영역 리디자인
- 필요 이상 리팩터링

## 3. 비교 절차

페이지 지정 후 아래 순서로 확인한다.

1. 현재 React 대응 페이지와 관련 테스트/클라이언트 코드를 확인한다.
2. PHP 원본의 페이지, JS, DB 엔드포인트를 확인한다.
3. 아래 항목으로 차이를 분류한다.

- `누락`: 원본 기능이 현재 없음
- `오개발`: 원본과 동작이 다름
- `스펙아웃 위반`: 원본에 없는데 현재 추가됨
- `구조적 예외`: 인증 변경 때문에 허용되는 차이
- `확인 필요`: 원본만으로 판단이 불명확함

## 4. 페이지별 완료 기준

한 페이지를 완료로 보려면 아래를 모두 충족해야 한다.

1. 원본 기능 목록이 문서화되어 있다.
2. 현재 구현과의 차이점이 근거 파일 경로와 함께 정리되어 있다.
3. 누락/오개발/스펙아웃 위반 항목이 코드로 반영되었다.
4. 인증 구조로 인한 예외 항목만 별도로 남아 있다.
5. 최소 검증 결과가 기록되어 있다.

## 9. 사용자 확인이 필요한 항목

아래는 페이지 작업 시작 전에 페이지별로 확인이 필요할 수 있다.

- 현재 실사용하지 않는 숨김 기능도 원복 대상인지

기본값은 다음과 같이 둔다.

- 모든 비교 기준은 PHP 원본으로 둔다.

### 확인 완료 사항

- `PHP 프로젝트 관리`는 유지 대상이다.
- 따라서 `pages/project.php` 기준 프로젝트 관리 기능은 복구 대상으로 처리한다.
- `트래킹`은 스펙아웃 대상이다.
- `pages/userset.php`의 PHP 비밀번호 변경 화면은 복구 대상이 아니며 현재 이관 코드를 유지한다.

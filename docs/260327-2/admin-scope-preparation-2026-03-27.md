## 0. 기준 문서 고정

2차 관리자 범위의 이관 기준 문서는 `docs/admin-analysis/README.md`로 고정한다.

- `docs/admin-analysis` 하위의 세부 문서는 `README.md` 해석을 보조하는 근거 문서로 사용한다.
- 1차 사용자 범위 문서와 충돌할 경우, 관리자 영역은 `docs/admin-analysis/README.md` 기준으로 정리한다.

## 1. 목적

이번 문서는 바로 구현에 들어가기 전, 2차 관리자 작업에서 다음 두 가지를 먼저 고정하기 위한 준비 문서다.

1. 1차와 겹치는 기능 중 무엇을 공용으로 잔존시키고, 무엇을 2차 작업에서 건드리지 말아야 하는지 정리
2. 2차 관리자 범위에서 스펙아웃 후보를 정리

핵심 원칙은 다음과 같다.

- 1차 작업 완료 범위는 건드리지 않는다.
- 2차 관리자 기능은 원칙적으로 2차 원본과 동일하게 맞춘다.
- 단, 인증 구조 변경, DB 구조 변경, 최소 버그 수정은 현재 이관 구조를 유지한다.
- 현재 페이지는 모두 각각의 독립 페이지로 구성하며, 탭 기반 화면 구성은 허용하지 않는다.
- 애매한 항목은 이번 문서에서 `사용자 확인 필요`로 남기고 임의 확정하지 않는다.

## 2. 이번 준비 작업의 해석 기준

현재 문서/사용자 지시 기준으로 2차 관리자 범위는 아래처럼 읽는다.

- `전체 업무검색`
- `사용자 관리`
- `프로필`
- `리소스 집계`
- `업무 타입 관리모드 CRUD`
- `서비스 그룹 관리모드 CRUD`
- `업무 타입 유효성검증`은 스펙아웃
- `서비스 그룹 유효성검증`은 스펙아웃
- `아지트 QA 알리미`는 스펙아웃

단, 원본 2차 코드에는 `업무 타입`, `서비스 그룹`의 `관리모드`와 `유효성검증모드`가 모두 존재한다.  
하지만 이번 작업에서는 사용자 지시를 우선 적용해 `유효성검증`만 스펙아웃으로 확정하고, `관리모드 CRUD`는 복원 대상으로 유지한다.

즉, 아래 항목은 이번 범위에서 제외다.

- `업무 타입 유효성검증`
- `서비스 그룹 유효성검증`
- `아지트 QA 알리미`

## 8. 페이지 작업 순서 제안

2차 관리자 작업은 아래 순서가 가장 안전하다.

1. `대시보드 병합 기준 정리`
2. `프로필`
3. `전체 업무검색`
4. `사용자 관리`
5. `업무 타입 관리모드 CRUD`
6. `서비스 그룹 관리모드 CRUD`
7. `리소스 집계`

이 순서를 제안하는 이유는 다음과 같다.

- 대시보드는 별도 2차 페이지가 아니라 1차 대시보드 병합 기준을 먼저 고정해야 이후 집계/관리 정보 배치를 판단할 수 있다.
- `프로필`, `전체 업무검색`, `사용자 관리`, `업무 타입/서비스 그룹 관리모드`, `리소스 집계`가 이번 2차 관리자 복원 핵심이다.
- 타입/서비스 `유효성검증`만 이번 범위에서 스펙아웃 확정이므로 작업 순서에서 제외한다.

## 11. 이번 준비 단계 결론

이번 단계 기준으로는 아래처럼 진행하는 것이 가장 안전하다.

- 1차와 겹치는 공용 데이터/기준정보는 `잔존`시키되, 1차 완료 범위는 사용자 승인 없이 수정하지 않는다.
- 대시보드는 1차 기능과 병합되는 공용 영역으로 관리한다.
- 2차 관리자 작업은 `프로필`, `전체 업무검색`, `사용자 관리`, `업무 타입 관리모드 CRUD`, `서비스 그룹 관리모드 CRUD`, `리소스 집계`를 복원 대상으로 잡는다.
- `타입/서비스 유효성검증`, `아지트 QA 알리미`는 스펙아웃 확정으로 유지한다.
- 각 기능은 독립 페이지로 구현하며 탭 구조는 허용하지 않는다.

다음 페이지 지정이 들어오면 이 문서를 기준으로 페이지별 대조 보고서와 수정 작업으로 바로 이어간다.

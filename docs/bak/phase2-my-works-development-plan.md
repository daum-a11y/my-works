# my-works 2차 개발 적용 계획서

작성일: 2026-03-24
기준 코드: `/my-works` 현재 구현 상태
관련 문서: `docs/transition/phase2-admin-review.md`, `my-works/STATUS.md`

## 1. 문서 목적

이 문서는 `/my-works` 현 코드 기준으로 1차 개발 완료 상태를 재정리하고, 2차 개발 범위를 어디에 어떻게 얹을지에 대한 적용 계획을 정의한다.

핵심 전제는 다음과 같다.

- 1차 개발은 이미 구현되어 있으며, 2차는 기존 사용자 기능 위에 관리자 축을 확장하는 작업이다.
- 2차는 레거시 관리자 기능 전체 복원이 아니라, 실제 필요한 운영 도구를 현재 React + Supabase 구조에 맞게 재구성하는 작업이다.
- `아지트 QA알리미`, `업무타입 관리모드`, `서비스 그룹 관리모드`는 이번 범위에서 제외한다.

## 2. 현재 구현 기준 요약

### 2.1 완료된 1차 범위

현재 `/my-works` 코드에는 아래 범위가 이미 구현되어 있다.

- 인증: Supabase Auth 기반 로그인, `members` 매핑, 비밀번호 변경
- 사용자 공통 화면: 대시보드, 개인 업무보고, 프로젝트 관리, 진행현황, 개인 검색, 통계
- 데이터 축: `members`, `task_types`, `service_groups`, `projects`, `project_pages`, `tasks`
- 저장 방식: Supabase RPC 중심 저장 (`save_task`, `delete_task`, 프로젝트/페이지 upsert 계열)

### 2.2 현재 코드에서 확인된 2차 미구현 지점

현재 2차와 직접 연결되는 갭은 아래와 같다.

1. 관리자 전용 라우트와 메뉴가 없다.
- `apps/ops-web/src/app/AppRouter.tsx`
- `apps/ops-web/src/app/AppShell.tsx`

2. 검색은 개인 검색만 구현되어 있다.
- `apps/ops-web/src/features/search/search-page.tsx`
- 현재 사용자 본인 `member` 기준 검색만 수행하며, 전체 업무보고 조회/수정/삭제/내보내기 기능이 없다.

3. 사용자 관리 화면이 없다.
- `members`, `new_member` 범위를 처리하는 관리자 UI와 전용 데이터 경로가 없다.

4. 업무타입/서비스그룹 검증 및 일괄 보정 흐름이 없다.
- 마스터 테이블은 존재하지만, 운영자가 데이터 불일치를 찾아 수정하는 보정 UI는 없다.

5. 내보내기 기능은 스캐폴드만 존재한다.
- `my-works/supabase/functions/export-tasks/index.ts`
- 현재는 실제 다운로드 기능이 아니라 placeholder 응답만 반환한다.

## 3. 2차 범위 재정의

`STATUS.md`의 2차 분리 범위(`allreport`, `type`, `service_group`, `members`, `new_member`)를 현재 사용자 결정사항까지 반영하여 아래처럼 재정의한다.

### 3.1 포함 범위

1. 전체 업무보고 관리자 검색
- 전체 사용자 대상 조회
- 기간/사용자/프로젝트/페이지/업무유형/서비스그룹 기준 필터링
- 관리자 수정/삭제
- 내보내기

2. 사용자 관리
- 사용자 목록 조회
- 권한/활성 상태/기본 정보 관리
- 인증 계정 연결 상태 확인

3. 신규 사용자 처리
- 별도 독립 시스템보다는 사용자 관리 내부의 onboarding queue로 해석
- `auth_user_id` 미연결, 이메일 매칭 대기, 권한 미설정 사용자를 운영자가 처리

4. 업무유형/서비스그룹 validation 및 bulk correction
- 마스터 CRUD가 아니라, 현재 데이터의 불일치 탐지와 일괄 보정
- 레거시 데이터 정리와 운영 품질 보정에 초점

### 3.2 제외 범위

1. 아지트 QA알리미
2. 업무타입 관리모드 CRUD
3. 서비스 그룹 관리모드 CRUD
4. 비활성 프로젝트 관리 UI
5. 추천 모니터링
6. 앱 운영정보
7. 알림 계열
8. `manager_*` 계열 레거시 범위

## 4. 적용 원칙

### 4.1 기존 1차 구조를 깨지 않는다

2차는 기존 사용자 동선을 갈아엎는 작업이 아니다. 현재 `/reports`, `/projects`, `/tracking`, `/reports/search` 흐름은 유지하고, 관리자 기능은 `/admin/*` 축으로 분리한다.

### 4.2 데이터 모델은 최대한 재사용한다

새 테이블을 성급하게 늘리기보다 현재 스키마를 우선 활용한다.

- `members`로 사용자 관리 처리
- `tasks`로 전체 업무보고 관리 처리
- `task_types`, `service_groups`는 참조 기준으로 활용
- 보정 기능은 신규 마스터 화면이 아니라 운영 보정 도구로 구현

### 4.3 쓰기 규칙은 RPC로 통제한다

현재 저장 구조가 RPC 중심이므로, 2차도 관리자 전용 조회/수정/보정 로직은 RPC 또는 DB function으로 모으는 편이 맞다. 클라이언트에서 테이블 직접 쓰기를 늘리면 권한 규칙과 감사 포인트가 분산된다.

### 4.4 화면 공용화는 억지로 하지 않는다

개인 검색과 전체 검색은 일부 필터 모델은 비슷하지만 권한, 결과 범위, 수정 책임이 다르다. 공통화는 `filter/result/query` 수준까지만 검토하고, 화면까지 억지로 합치지는 않는다.

## 5. 상세 개발 계획

## 5.1 작업축 A: 관리자 라우트와 권한 셸 구성

### 목표

관리자 전용 화면 진입 구조를 먼저 만든다.

### 적용 대상

- `apps/ops-web/src/app/AppRouter.tsx`
- `apps/ops-web/src/app/AppShell.tsx`
- `apps/ops-web/src/features/auth/AuthContext.tsx`

### 적용 방식

1. `/admin/reports`
2. `/admin/members`
3. `/admin/validation`

위 3개를 2차 기본 라우트로 추가한다.

권한 처리는 현재 `AuthContext`의 사용자/멤버/role 정보를 기준으로 처리한다.

- `admin`만 관리자 라우트 접근 허용
- 일반 사용자는 기존 1차 메뉴만 노출
- 관리자도 기존 사용자 화면 접근은 유지

### 산출물

- 관리자 메뉴 노출 규칙
- 관리자 라우트 가드
- 관리자 화면 기본 레이아웃 또는 AppShell 분기

## 5.2 작업축 B: 전체 업무보고 관리자 검색

### 목표

레거시 `allreport` 성격의 기능을 현재 구조에 맞게 재구성한다.

### 현재 기반

- 개인 검색: `apps/ops-web/src/features/search/search-page.tsx`
- 업무 저장/삭제: `apps/ops-web/src/lib/data-client.ts`, DB RPC
- 업무 도메인: `apps/ops-web/src/lib/domain.ts`, `reports` 관련 코드

### 구현 범위

1. 전체 사용자 대상 검색
2. 다중 조건 필터
3. 결과 목록 정렬 및 페이징
4. 관리자 수정
5. 관리자 삭제
6. 내보내기 호출

### 권장 구현 방식

- 관리자 전용 검색 함수 추가
- 예시: `searchTasksAdmin(filters)` 또는 동등 기능의 RPC wrapper
- 조회는 관리자 전용 SQL/RPC로 분리
- 수정/삭제는 기존 `save_task`, `delete_task` 재사용 가능 여부를 먼저 확인하고, 관리자 권한 분기만 보강

### 화면 구조 제안

1. 상단 필터 바
- 기간
- 사용자
- 프로젝트
- 페이지
- 업무유형
- 서비스그룹
- 키워드

2. 본문 결과 테이블
- 작성일
- 사용자
- 프로젝트/페이지
- 업무유형
- 처리 상태/시간
- 수정/삭제 액션

3. 우측 또는 하단 상세 패널
- 행 선택 시 상세 보기 및 수정

### 비고

개인 검색 페이지를 그대로 확장하기보다, 관리자 검색 페이지를 신규 생성하고 일부 필터/결과 표현만 재사용하는 편이 안전하다.

## 5.3 작업축 C: 사용자 관리와 신규 사용자 처리

### 목표

`members`, `new_member` 범위를 현재 스키마 기반 운영도구로 정리한다.

### 현재 기반

- `members` 테이블 존재
- 로그인 시 `auth_user_id` 또는 이메일 기준 매핑 수행
- 별도 관리자 UI는 없음

### 구현 범위

1. 사용자 목록 조회
2. 이름/이메일/부서/권한/활성 여부 편집
3. 인증 계정 연결 상태 확인
4. 신규 사용자 후보 처리

### 신규 사용자 처리 해석

별도 `new_member` 독립 화면보다, 사용자 관리 화면 내에 아래 큐를 두는 방식이 현실적이다.

- `auth_user_id` 미연결 사용자
- 이메일 불일치 사용자
- role 미설정 사용자
- 운영상 비활성 전환 후보

### 권장 구현 방식

- `listMembersAdmin`
- `saveMemberAdmin`
- 필요 시 `linkMemberAuthAdmin` 또는 동등 기능의 RPC 추가

클라이언트는 `members` 테이블 직접 수정보다 RPC를 우선 사용한다.

## 5.4 작업축 D: 업무유형/서비스그룹 validation 및 bulk correction

### 목표

관리모드 CRUD 대신, 실제 데이터 정합성 보정 도구를 만든다.

### 왜 이 축이 필요한가

현재 스키마에는 `task_types`, `service_groups` 마스터가 있지만, 업무 데이터 입력과 레거시 이관 데이터가 모두 동일한 기준으로 정규화되어 있다고 보기 어렵다. 2차에서 필요한 것은 마스터 관리 화면보다 운영자가 불일치를 찾아 정리할 수 있는 도구다.

### 구현 범위

1. 업무유형 미매핑/오타/구형 값 탐지
2. 서비스그룹 불일치 탐지
3. 기준값 대비 invalid 데이터 목록화
4. 일괄 치환 또는 일괄 매핑
5. 보정 결과 재조회

### 권장 구현 방식

- 읽기 전용 기준 목록: `task_types`, `service_groups`
- 검증 쿼리: invalid value 집계, 영향 건수, 최근 사용일
- 일괄 보정 RPC: source value -> target value 매핑 적용

### 화면 구조 제안

1. validation summary
- invalid 업무유형 수
- invalid 서비스그룹 수
- 영향 task 수

2. issue list
- 원본 값
- 후보 기준값
- 영향 건수
- 최근 발생일

3. bulk action
- 대상 선택
- 치환 기준값 선택
- 적용

### 명시 사항

이 화면은 마스터 등록/수정/삭제 화면이 아니다. 기준값 변경이 아니라, 잘못 저장된 운영 데이터를 기준값에 맞추는 보정 화면이다.

## 5.5 작업축 E: 내보내기 기능 완성

### 목표

현재 placeholder 상태인 `export-tasks`를 실제 운영 기능으로 바꾼다.

### 현재 기반

- `my-works/supabase/functions/export-tasks/index.ts`
- 현재는 샘플 CSV 문자열 반환 수준

### 권장 구현 방식

1. 관리자 검색 필터와 동일한 조건으로 export 요청
2. Edge Function에서 관리자 권한 검증
3. CSV 생성 후 다운로드 응답 반환

### 형식 전략

실무 기준으로는 우선 `CSV(Excel 호환)`로 시작하는 것이 맞다.

- UTF-8 BOM 포함
- 헤더/값 escaping 처리
- 날짜/시간/작성자/프로젝트/페이지/업무유형/소요시간 포함

서식형 `.xlsx`가 반드시 필요한 요구가 확정되기 전까지는 CSV가 구현 복잡도와 운영 효율 측면에서 적절하다.

## 7. 권장 개발 순서

### 1단계: 관리자 진입 구조

- 관리자 라우트 추가
- 메뉴/권한 가드 추가
- 빈 관리자 페이지 3종 구성

### 2단계: 전체 업무내역 조회 + 수정/삭제

- 관리자 검색 API/RPC 추가
- 관리자 결과 테이블 구현
- 수정/삭제 연결

### 3단계: 내보내기

- 검색 필터 연동 export
- CSV 다운로드 완성

### 4단계: 사용자 관리

- 사용자 목록/상세/저장
- 신규 사용자 처리 큐 반영

### 5단계: validation / bulk correction

- invalid 데이터 집계
- 일괄 치환 기능
- 보정 후 재조회

이 순서가 적절한 이유는, 2차에서 가장 먼저 운영 가치가 발생하는 기능이 전체 업무내역 조회과 관리자 수정이기 때문이다. 사용자 관리와 validation은 그 다음 축이다.

## 8. 리스크와 확인 필요사항

### 8.1 데이터 정규화 수준 확인 필요

현재 업무유형/서비스그룹이 어느 수준까지 master key 기반으로 정규화되어 있는지 추가 확인이 필요하다. 이 값이 text 혼합 상태라면 validation/bulk correction 우선순위가 더 올라간다.

### 8.2 관리자 수정 범위 확정 필요

관리자가 전체 업무보고에서 어디까지 수정 가능한지 확정이 필요하다.

- 본문만 수정 가능한지
- 프로젝트/페이지 재지정까지 가능한지
- 작성자 변경까지 허용할지

이 범위에 따라 `save_task` 재사용 범위가 달라진다.

### 8.3 신규 사용자 정의 확정 필요

`new_member`가 별도 엔터티인지, 아니면 가입 후 미연결 사용자를 뜻하는 운영 큐인지 레거시 의미 확인이 필요하다. 현재 코드 기준으로는 별도 테이블 신설보다 운영 큐 해석이 더 자연스럽다.

### 8.4 대용량 export 처리 한계

건수가 많아지면 동기식 export 응답이 무거워질 수 있다. 초기 2차는 동기 CSV 응답으로 시작하고, 필요 시 비동기 파일 생성으로 분리하는 것이 적절하다.

## 9. 완료 기준

아래 조건을 만족하면 `/my-works` 기준 2차 핵심 범위가 적용된 것으로 본다.

1. 관리자 로그인 시 관리자 메뉴가 노출된다.
2. 관리자는 전체 사용자 업무보고를 검색할 수 있다.
3. 관리자는 검색 결과를 수정/삭제할 수 있다.
4. 관리자는 검색 결과를 CSV로 내보낼 수 있다.
5. 관리자는 사용자 정보를 조회/수정할 수 있다.
6. 관리자는 신규 사용자 처리 대상을 식별하고 상태를 정리할 수 있다.
7. 관리자는 업무유형/서비스그룹 불일치를 조회하고 일괄 보정할 수 있다.
8. 기존 1차 사용자 화면 동선은 유지된다.

## 10. 결론

현재 `/my-works`는 1차 사용자 기능이 이미 상당 부분 정리되어 있으므로, 2차는 새 앱을 다시 만드는 작업이 아니다. 필요한 것은 현재 코드 위에 관리자 운영축을 분리해서 얹는 일이다.

가장 우선순위가 높은 2차 구현은 아래 3개다.

1. 관리자 전체 업무내역 조회/수정/삭제
2. export 완성
3. 사용자 관리

그 다음으로 업무유형/서비스그룹 validation 및 bulk correction을 붙이면, 레거시 관리자 기능을 현재 구조에 맞는 형태로 무리 없이 정리할 수 있다.

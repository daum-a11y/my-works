# 현행 버전 임시 운영 및 2.0 전환 검토 보고서

## 1. 개요

본 문서는 `php-operation_tool` 현행 시스템을 기존 클라우드에서 더 이상 유지하기 어려운 상황에서, 2.0 배포 전까지 사용할 대체 버전의 구현 방향을 검토한 보고서입니다.

이번 검토의 기준은 아래 한 줄입니다.

`이번 계획은 현재 기능을 그대로 React 기반으로 옮기는 것이다.`

즉, 이번 과제는 기능 축소 브리지를 새로 정의하는 작업이 아니라, 현행 운영 기능을 다른 런타임과 보안 구조 위에 재구현하는 작업으로 봐야 합니다.

## 2. 배경

- 현행 시스템은 PHP 7.1 기반 레거시 운영툴입니다.
- 인증, 권한, 입력 검증, 비밀정보 관리 수준이 낮아 보안 검수가 어렵습니다.
- 클라우드 정책 변경으로 기존 형태의 PHP 재배포가 어렵습니다.
- 2.0은 진행 중이지만 2026년 3월 말까지 개발 및 QA 완료가 어렵습니다.
- 2026년 4월부터는 기존 클라우드 서버를 내려야 합니다.
- 따라서 2.0 전까지 사용할 대체 운영 버전이 필요합니다.

## 3. 이번 계획의 해석

이번 계획은 다음처럼 해석하는 것이 맞습니다.

1. 현행 운영 기능은 그대로 유지한다.
2. 다만 PHP가 아니라 React 기반 클라이언트로 재구현한다.
3. 인증/보안/저장 검증은 현재보다 확실히 개선한다.
4. 이후 2.0 도입 시 데이터와 업무 흐름이 자연스럽게 넘어가야 한다.

따라서 이번 단계에서 `과거 자료는 검색만 가능`을 기본 전략으로 두는 것은 맞지 않습니다.

정확한 위치는 다음과 같습니다.

- 현재 임시 버전: 현행 기능 그대로 운영
- 향후 2.0 이행: 일부 오래된 이력 데이터가 정말 이관 난도가 높을 때만 예외적으로 검색 전용 fallback 검토

## 4. 현행 시스템의 핵심 문제

### 4.1 보안과 인증

- 자체 세션 기반 로그인
- 하드코딩된 DB 접속 정보
- 문자열 결합 SQL
- 엔드포인트별 불균일한 권한 체크

핵심은 `PHP라서 문제`가 아니라 `현재 인증/보안 구조가 현행 운영 기준을 통과하기 어렵다`는 점입니다.

### 4.2 구현 구조

- `pages/*.php`, `dbcon/*.php`, jQuery fragment 응답 구조
- `TASK_TBL` 중심의 텍스트 복제 저장
- 숨김 화면, 호출처 불명 코드, 레거시 잔존 스펙 혼재

즉, 화면만 옮기면 끝나는 작업이 아니라 저장 규칙과 권한 규칙을 같이 정리해야 합니다.

### 4.3 데이터 품질

- 등록 시 일부 컬럼이 `NULL` 되는 버그가 있다고 공유됨
- 현재 구조상 입력 검증과 저장 규칙이 느슨해 재현 가능성이 높음

따라서 이번 이관에서 가장 먼저 바로잡아야 할 것은 UI가 아니라 `저장 계약`입니다.

## 5. 목표

이번 임시 버전의 목표는 다음입니다.

- 현행 기능 동일 수준 운영 가능
- 인증/권한/입력 검증 강화
- 기존 데이터 운영 연속성 유지
- 이후 2.0로 마이그레이션 가능한 구조 확보

## 6. 권장 아키텍처 방향

### 6.1 Supabase 중심 기본안

`React CSR + Supabase Auth + Supabase Postgres + RLS + RPC/Edge Functions`

지금 설명하신 의도대로라면, 이번 단계의 가장 자연스러운 기본안은 이 구조입니다.

핵심은 `별도 API 서버를 운영하지 않는다`는 점보다, `Supabase를 앱 백엔드로 사용한다`는 점입니다.

즉:

- 인증: Supabase Auth
- 데이터: Supabase Postgres
- 단순 CRUD: Supabase direct query
- 복잡한 저장 규칙: RPC 또는 Edge Functions

이 구성이 맞는 이유:

- 어차피 API 의미를 다시 설계해야 한다면, 별도 Node API를 한 번 더 두는 것보다 Supabase 안으로 모으는 편이 단순합니다.
- 인하우스 DB 접근 승인과 보안검수 리드타임을 피할 수 있습니다.
- 로그인/세션/토큰/기본 권한 체계를 같이 가져갈 수 있습니다.
- 임시 버전이더라도 2.0에서 재사용 가능한 인증/DB 축을 먼저 깔 수 있습니다.

### 6.2 최소 성립안

현재 상황에서 가장 현실적인 최소 성립안은 다음입니다.

- `USER_TBL` 성격의 멤버 정보를 Supabase 테이블로 이관
- `members.auth_user_id`는 nullable 허용
- Supabase Auth 계정과 멤버 레코드를 1:1 바인딩
- 로그인 후 바인딩된 멤버 정보로 권한 판단
- 일반 CRUD는 Supabase direct query
- 다중 테이블 갱신이나 저장 규칙이 복잡한 케이스만 RPC/Edge Functions

즉, 인증 보강 자체는 큰 작업이 아닙니다.

실제로 필요한 것은 아래 정도입니다.

- `auth.users.id` 와 멤버 레코드 바인딩
- `legacy user_id` 보존
- `user_level`, `user_active`, `report_required` 같은 필드 매핑
- 로그인한 사용자의 legacy id를 기준으로 기존 업무 데이터 연결
- 초기 데이터 적재와 계정 생성 시점을 분리

권장 흐름:

1. 기존 `USER_TBL` 기준으로 `members`를 먼저 일괄 적재
2. `auth_user_id`는 비워둔 상태로 운영 준비
3. 실제 계정이 생성되면 스크립트나 관리 SQL로 후행 바인딩
4. 바인딩된 사용자만 로그인 허용

즉, 계정 생성이 실제 데이터보다 늦게 생길 수 있는 상황을 전제로 잡아야 합니다.

지금처럼 `인증이랄 게 거의 없는` 상태라면, 말씀하신 대로 멤버 정보만 Supabase 계정과 바인딩해도 바로 실사용 가능한 수준까지 갈 수 있습니다.

### 6.3 중요한 구분

다만 `Supabase에 직접 콜한다`는 말을 아래처럼 구분해야 합니다.

허용 가능한 direct call:

- 본인 데이터 조회
- 목록 조회
- 단순 등록/수정/삭제
- 권한이 명확한 CRUD

직접 raw table call로 끝내면 위험한 영역:

- 업무보고 저장 시 복수 컬럼 규칙 검증
- 프로젝트/페이지 수정 시 연관 데이터 정합성 처리
- 등록 시 null 방지
- 다운로드 생성
- 관리자 전용 기능

이 영역은 별도 Node API가 아니라 `Supabase RPC/Edge Functions`로 처리하는 것이 맞습니다.

즉, 제 말의 핵심은 `Node API를 반드시 따로 둬야 한다`가 아니라, `업무 규칙과 저장 검증을 브라우저 코드에 두면 안 된다`는 것이었습니다.

### 6.4 MySQL 재사용안은 언제 유리한가

`React CSR + Node.js API + 기존 MySQL 재사용`

이 안은 다음 조건이면 더 빠를 수 있습니다.

- 기존 MySQL 연결이 쉽고
- 인하우스 접근 승인 리드타임이 짧고
- DB 이전 자체를 최대한 피하고 싶고
- 정말 1~2개월 임시 사용 후 폐기 가능성이 높은 경우

하지만 현재 설명처럼 인하우스 DB 접근권한과 보안검수가 병목이면, 이 안은 개발 MD는 낮아도 실제 일정은 불리해질 수 있습니다.

### 6.5 운영 리드타임 기준의 예외

만약 아래 항목이 붙으면 판단이 달라집니다.

- 인하우스 DB 접근 권한 신청
- 별도 보안검수
- 네트워크 화이트리스트/망연결 승인
- 운영계정 발급 및 접속 검증

이 경우 `기존 MySQL 재사용`은 개발 공수는 낮아도 실제 오픈 일정은 가장 불확실해질 수 있습니다.

즉, 개발 MD만 보면 MySQL 재사용이 유리할 수 있지만, `실제 배포 가능 시점`까지 포함하면 Supabase 쪽이 오히려 단순해질 수 있습니다.

## 7. 데이터 전략

### 7.1 이번 단계의 원칙

이번 임시 버전에서는 원칙적으로 `검색 전용 분리`를 기본 전략으로 두지 않습니다.

이유:

- 계획 자체가 `현행 기능 그대로 이관`이기 때문입니다.
- 운영 기능을 동일하게 유지하려면 기존 데이터도 동일한 수준으로 읽고 써야 합니다.
- 검색 전용 전환은 기능 축소안일 뿐, 현재 합의된 방향과 다릅니다.

### 7.2 이번 단계의 권고

- 현행 기능은 동일하게 유지
- 인증은 Supabase Auth 사용
- 멤버 정보는 Supabase 계정과 바인딩
- 초기 적재 시 `auth_user_id` 없이 멤버와 업무 데이터를 먼저 올릴 수 있게 설계
- 데이터 저장소는 Supabase Postgres 사용
- 저장 규칙은 현행 MySQL 해석을 최대한 유지
- 단순 조회는 direct query
- 복잡한 저장은 RPC/Edge Functions

즉, 이번 단계는 `UI 재구현 + 백엔드 전환`이지만, `업무 규칙 해석`은 최대한 그대로 두는 방식이 맞습니다.

### 7.3 향후 2.0 이행 시

향후 2.0 전환에서 다음 경우에만 검색 전용 fallback을 검토할 수 있습니다.

- 오래된 종료 이력 중 수정 필요성이 거의 없는 데이터
- 정규화 비용이 매우 큰 레거시 텍스트 복제 이력
- 실운영 기능과 직접 연결되지 않는 보조 데이터

즉, 검색 전용은 현재 임시 버전의 원칙이 아니라, 향후 2.0 전환 시 일부 과거 데이터에 대한 예외 전략입니다.

## 8. 동일 이관 대상 범위

현재 계획 기준으로는 다음을 기본 이관 범위로 보는 것이 맞습니다.

### 8.1 1차 동일 이관 대상

- 로그인/로그아웃
- 사용자 권한 구분
- 업무보고 등록/수정/삭제/조회
- 프로젝트 등록/수정/조회
- 프로젝트 페이지 등록/수정/조회
- 트래킹 관리
- 개인 업무보고 검색/다운로드
- 대시보드
- QA/모니터링 통계
- 비밀번호 변경

### 8.2 2차 동일 이관 대상

- 앱 운영정보
- 전체 업무보고 화면
- Type 관리
- Service Group 관리
- 멤버 관리/신규 멤버 생성

### 8.3 별도 판단 대상

아래는 현재도 실제 동작 근거가 약하므로, "현행 기능 그대로" 범위와는 별개로 판단해야 합니다.

- `noti.php` 기반 실시간 알림
- 호출처 불명 `manager_*`
- 현재 저장소와 DB 구조가 어긋나는 알림 영역

즉, `지금도 불완전한 레거시 꼬리`까지 무조건 복제하는 것은 다른 문제입니다.

## 9. 구현 원칙

- 화면은 React로 재구현
- 인증은 Supabase Auth 사용
- 권한은 바인딩된 멤버 정보와 RLS로 판단
- `members.auth_user_id`는 nullable 허용 후 후행 연결
- 단순 CRUD는 Supabase direct query 사용
- 복잡한 저장 규칙은 RPC/Edge Functions로 처리
- 비밀정보는 Supabase 프로젝트 설정과 환경변수로 분리
- 등록 시 필수 컬럼 null 방지를 위해 입력 스키마와 저장 규칙을 명시

- React + Supabase

## 11. 공수 산정

### 11.1 시나리오별 추정

| 시나리오 | 범위 | 개발 공수 | QA/안정화 | 총 공수 |
| --- | --- | --- | --- | --- |
| A | Supabase Auth/DB + 메뉴 노출 기능 동일 이관 | 18~24 MD | 5~7 MD | 23~31 MD |
| B | A 범위 + 2차 관리 화면 동일 이관 | 22~30 MD | 6~8 MD | 28~38 MD |
| C | B 범위 + 불명확 레거시 꼬리까지 최대한 복원 | 30~42 MD | 7~10 MD | 37~52 MD |

위 수치는 `애플리케이션 개발 공수` 기준입니다. 인하우스 DB 접근 승인 리드타임은 별도 변수입니다.

리드타임 관점의 해석:

- `기존 MySQL 재사용`: 개발 MD는 낮을 수 있으나 운영 승인 일정이 불확실
- `Supabase 사용`: 개발 MD는 소폭 늘 수 있으나 운영 승인 경로가 단순

### 11.2 작업별 감각

| 작업 | 추정 공수 |
| --- | --- |
| Supabase Auth + 멤버 초기 적재/후행 바인딩 | 1~2 MD |
| 최소 RLS/권한 정책 | 1~2 MD |
| 인증/권한 | 3~5 MD |
| 공통 레이아웃/라우팅 | 2~3 MD |
| 업무보고 화면 및 저장 규칙 | 6~9 MD |
| 프로젝트/페이지/트래킹 | 5~8 MD |
| 검색/다운로드 | 2~4 MD |
| 대시보드/통계 | 3~5 MD |
| 관리 화면군 | 3~6 MD |
| API 검증/에러처리/null 버그 보강 | 4~6 MD |
| 배포/운영 설정 | 2~4 MD |
| QA 대응/안정화 | 5~8 MD |

### 11.3 해석

이 기준에서 보면, 제가 앞서 제시한 `50 MD`는 범위를 잘못 잡은 값이었습니다.

정확히는:

- `Supabase Auth/DB + 현행 기능 동일 이관` 기준의 본체는 대체로 `23~38 MD` 범위가 맞습니다.
- `40 MD` 이상으로 커지는 경우는 불명확 레거시 꼬리까지 같이 살리거나, 이번 단계에서 업무 규칙 자체를 새로 정리하려는 경우입니다.

중요한 점:

- 인증 보강 자체가 큰 것이 아닙니다.
- 지금 공수의 본체는 인증이 아니라 `현행 화면 재구현`, `저장 규칙 반영`, `기존 예외 로직 재현`입니다.
- 따라서 사용자 말씀처럼 `멤버 정보만 Supabase 계정과 바인딩하면 바로 쓸 수 있다`는 평가는 인증 영역만 놓고 보면 맞습니다.

즉, 지금 계획이 정말 `현재 기능 그대로 옮긴다`라면, 기준 공수는 `A~B 시나리오`로 봐야 합니다.

## 12. 일정 감각

- 2명 이상 병렬 투입이 사실상 필요합니다.
- 특히 업무보고/프로젝트/트래킹 쪽과 인증/데이터 적재 쪽을 병렬로 나눌 수 있어야 합니다.

## 13. 최종 판단

현재 합의된 계획을 기준으로 보면, 이번 과제의 가장 자연스러운 정의는 다음입니다.

`PHP 레거시를 React + Supabase 기반으로 동일 기능 이관하되, 인증은 Supabase Auth로 단순화하고 저장 규칙만 서버측 함수로 보완한다.`

이 기준에서의 권고는 다음과 같습니다.

1. 지금 설명 기준으로는 `React CSR + Supabase Auth/DB + RLS + RPC/Edge Functions`가 가장 일관된 기본안입니다.
2. 이 구조는 `직접 콜`을 쓰되, 복잡한 저장 규칙은 Supabase 서버측 기능으로 올리는 방식이어야 합니다.
3. `검색 전용`은 지금 임시 버전의 기본 전략이 아니라, 향후 2.0 이행 시 일부 과거 데이터에 대한 예외 fallback으로만 다뤄야 합니다.
4. 공수는 `메뉴 노출 기능만`이면 대체로 `23~31 MD`, `관리 화면 포함`이면 `28~38 MD` 수준으로 보는 것이 합리적입니다.
5. 이 범위가 2.0 최소 오픈 공수와 비슷해지면, 그때는 임시 버전보다 `2.0 최소 범위 오픈`이 더 맞습니다.

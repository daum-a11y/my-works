# 현행 버전 React 이관 작업계획서

## 1. 문서 목적

본 문서는 [`interim-react-migration-review.md`](./interim-react-migration-review.md)를 기준으로, `php-operation_tool`의 임시 운영 버전을 React 기반으로 이관하기 위한 실행 계획을 정리한 작업계획서입니다.

- 기준일: 2026-03-24
- 계획 기준: 실제 사용하는 메뉴와 업무 흐름 우선 마이그레이션
- 권장 기술축: `React CSR + Supabase Auth + Supabase Postgres + RLS + RPC/Edge Functions`
- 계획 목적: 2.0 전까지 운영 가능한 임시 버전을 현실적인 범위와 일정으로 오픈

본 계획은 검토 보고서의 `현행 기능 동일 이관` 원칙을 출발점으로 삼되, 실제 실행 범위는 `현재 실사용 중인 화면과 업무 흐름`에 한정해 빠르게 오픈하는 방향으로 조정합니다.

## 2. 추진 원칙

### 2.1 범위 원칙

- 이번 단계의 기본 목표는 `저장소 전체 복제`가 아니라 `실사용 기능 우선 마이그레이션`입니다.
- 현재 메뉴에 있더라도 실사용 근거가 약한 기능은 제외할 수 있습니다.
- 다만 현재도 동작 근거가 약한 레거시 꼬리는 본 계획의 기본 범위에서 제외합니다.
- 1차 오픈 범위와 2차 보강 범위를 명확히 나눠 일정 리스크를 통제합니다.

### 2.2 구현 원칙

- 화면은 React로 재구현합니다.
- 인증은 Supabase Auth를 사용합니다.
- 권한은 `members` 바인딩 정보와 RLS 기준으로 판단합니다.
- 단순 CRUD는 direct query로 처리합니다.
- 저장 규칙, 다중 테이블 정합성, null 방지, 관리자 기능, 다운로드 생성은 RPC 또는 Edge Functions로 처리합니다.
- 레거시 식별자와 업무 규칙 해석은 가능한 한 유지합니다.

### 2.3 데이터 원칙

- 검색 전용 fallback은 이번 임시 버전의 기본 전략으로 두지 않습니다.
- `members.auth_user_id`는 nullable 허용 상태로 먼저 적재합니다.
- 기존 `USER_TBL`의 `legacy user_id`를 보존해 업무 데이터 연결의 기준값으로 사용합니다.
- 초기 데이터 적재와 실제 계정 바인딩 시점은 분리합니다.

## 3. 범위 정의

### 3.1 1차 오픈 범위

1차 오픈은 검토 보고서의 `8.1 1차 동일 이관 대상`을 기준으로 구성합니다.

- 로그인/로그아웃
- 사용자 권한 구분
- 업무보고 등록/수정/삭제/조회
- 프로젝트 등록/수정/조회
- 프로젝트 페이지 등록/수정/조회
- 트래킹 관리
- 개인 업무내역 조회/다운로드
- 대시보드
- QA/모니터링 통계
- 비밀번호 변경

대시보드는 전체를 기계적으로 복제하지 않고, 아래 범위만 1차 포함 대상으로 봅니다.

- 진행중 모니터링 목록
- 진행중 QA 목록

아래 항목은 대시보드 범위에서 제외합니다.

- 추천 모니터링 iOS 앱 패널
- 추천 모니터링 Android 앱 패널
- `APPINFO_TBL` 기반 추천 모니터링 계산 로직

### 3.2 2차 보강 범위

2차 보강은 `별도 관리자 툴` 기준으로 분리합니다. 1차 사용자 화면 오픈과는 분리해, 2026년 4월 중순 이후 착수 가능한 범위로 둡니다.

- 전체 업무보고 화면 (`allreport.php`)
- Type 관리 (`type.php`)
- Service Group 관리 (`service_group.php`)
- 멤버 관리/신규 멤버 생성 (`members.php`, `new_member.php`)

정리 기준:

- 1차 범위는 현재 실사용 메뉴와 운영 핵심 흐름에 한정합니다.
- 관리자 기능은 동일 앱 안에 무리하게 끼워 넣지 않고, 별도 관리자 툴 또는 별도 관리자 영역으로 분리하는 것을 기본안으로 둡니다.
- 2차 범위의 상세 포함 방식은 1차 오픈 안정화 이후 확정합니다.

### 3.3 별도 판단 범위

아래 항목은 기본 범위에 포함하지 않고 별도 의사결정 항목으로 둡니다.

- 대시보드 추천 모니터링 화면
- 앱 운영정보 화면
- `APPINFO_TBL` 및 추천 모니터링 보조 데이터
- `noti.php` 기반 실시간 알림
- 호출처 불명 `manager_*`
- 저장소와 DB 구조가 어긋나는 알림 영역

## 4. 권장 추진 방식

### 4.1 권장 시나리오

현 시점의 일정 제약과 보고서의 공수 추정을 함께 보면, 본 계획은 `A 시나리오 선오픈 + B 시나리오 후속 반영`으로 추진하는 것이 가장 현실적입니다.

- 1차 목표: 실제 사용 중인 메뉴와 업무 흐름만 우선 오픈
- 2차 목표: 별도 관리자 툴 범위를 2026-04 중순 이후 순차 반영
- 제외 원칙: 불명확 레거시 꼬리는 오픈 임계경로에 넣지 않음

### 4.2 인력 전제

- 최소 필요 인원: 개발 2명
- 권장 인원: 개발 2명 + QA 1명, 또는 개발 3명
- 1명 단독 진행은 2026-03-24 기준 일정상 비현실적입니다.

### 4.3 권장 역할 분담

- 트랙 A: 인증, 권한, Supabase 스키마, 데이터 적재, RPC/Edge Functions
- 트랙 B: 공통 레이아웃, 라우팅, 업무보고, 프로젝트/페이지/트래킹 화면
- 트랙 C: 대시보드, 통계, 다운로드, 관리 화면, 통합 QA 대응

## 6. 단계별 계획

### 6.1 Phase 0. 착수 및 기준선 확정

목적은 오픈 임계경로에 들어갈 범위를 확정하고, 레거시 해석 차이로 인한 재작업을 줄이는 것입니다.

- 메뉴 단위로 1차 범위와 2차 범위를 확정합니다.
- 현재도 불명확한 `noti`, `manager_*`, 알림 영역을 예외 항목으로 고정합니다.
- 기능별 완료 기준을 `조회`, `등록`, `수정`, `삭제`, `권한`, `다운로드` 단위로 분해합니다.
- 현행 저장 규칙 중 반드시 서버측 함수로 올려야 할 케이스를 우선 식별합니다.

산출물:

- 메뉴-기능-권한 매핑표
- 범위 제외 목록
- 저장 계약 우선순위 목록

### 6.2 Phase 1. Supabase 기반 구축

목적은 인증과 데이터 축을 먼저 안정화하는 것입니다.

- Supabase Auth 구성
- `members` 테이블 설계 및 `auth_user_id` nullable 구조 반영
- `legacy user_id`, `user_level`, `user_active`, `report_required` 필드 매핑
- 초기 멤버 적재 및 후행 계정 바인딩 절차 확립
- 최소 RLS 정책과 관리자/일반 사용자 접근 경계 설정

산출물:

- 초기 스키마
- 멤버 적재 스크립트 또는 SQL
- 계정 바인딩 운영 절차
- RLS 정책 초안

### 6.3 Phase 2. 저장 계약과 서버 함수 구현

목적은 UI보다 먼저 저장 정합성과 null 방지를 통제하는 것입니다.

- 업무보고 저장 규칙 정의
- 프로젝트/페이지 수정 시 연관 데이터 정합성 규칙 정의
- 입력 스키마와 필수값 검증 규칙 정의
- 다운로드 생성 규칙 정의
- 관리자 전용 저장 로직을 RPC 또는 Edge Functions로 구현

산출물:

- 도메인별 저장 계약 문서
- RPC/Edge Functions
- 입력 검증 스키마

### 6.4 Phase 3. 1차 화면 구현

목적은 1차 오픈 대상 기능을 실제 운영 가능한 수준으로 재구현하는 것입니다.

- 로그인/로그아웃, 비밀번호 변경
- 공통 레이아웃과 라우팅
- 업무보고 등록/수정/삭제/조회
- 프로젝트/페이지 등록/수정/조회
- 트래킹 관리
- 개인 업무내역 조회/다운로드
- 대시보드(진행중 모니터링 목록, 진행중 QA 목록만 포함)
- QA/모니터링 통계

산출물:

- 1차 메뉴 전체 UI
- 화면별 데이터 연동
- 공통 에러/권한 처리

### 6.5 Phase 4. 통합 검증과 오픈 준비

목적은 실사용 기준의 회귀와 데이터 정합성을 검증하는 것입니다.

- 사용자 권한별 시나리오 검증
- 기존 대표 데이터 기준 조회 결과 대조
- 입력 필수값, null 방지, 저장 후 재조회 검증
- 다운로드 결과 검증
- 운영 전환 체크리스트 점검

산출물:

- 통합 테스트 체크리스트
- 결함 수정 목록
- 오픈 전환 체크리스트

### 6.6 Phase 5. 2차 관리자 툴 구현

1차 오픈 이후 안정화가 끝나면, 관리자 전용 기능은 별도 관리자 툴 기준으로 분리 구현합니다. 착수 시점은 `2026-04-13` 이후를 기본안으로 둡니다.

- 전체 업무보고
- Type 관리
- Service Group 관리
- 멤버 관리/신규 멤버 생성

산출물:

- 관리자 툴 UI 및 서버 로직
- 관리자 권한 검증 시나리오

## 7. 우선순위와 병렬화 계획

### 7.1 최우선 작업

아래 항목은 오픈 임계경로이므로 가장 먼저 병렬 착수해야 합니다.

1. `members` 바인딩 구조와 권한 정책 확정
2. 업무보고 저장 계약 정리
3. 공통 앱 셸과 인증 가드 구성
4. 프로젝트/페이지/트래킹 저장 규칙 확정

### 7.2 병렬 작업 구조

- 개발자 A: 인증, 권한, 스키마, 적재, 서버 함수
- 개발자 B: 공통 UI, 업무보고, 프로젝트/페이지/트래킹
- QA 또는 개발자 C: 통계, 다운로드, 회귀 체크리스트, 관리 화면 준비

### 7.3 후순위 작업

아래 항목은 1차 오픈 이후 또는 여유 시점에만 반영합니다.

- 2차 관리자 툴
- 불명확 레거시 꼬리
- 구조가 맞지 않는 알림 기능

## 8. 예상 공수와 일정안

### 8.1 재산정 기준

검토 보고서의 기존 기준선은 아래와 같았습니다.

- A 시나리오: `23~31 MD`
- B 시나리오: `28~38 MD`

현재 작업계획서에서는 아래 범위를 제외했으므로, 기존 기준선을 그대로 쓰지 않고 소폭 하향해 재산정합니다.

- 대시보드 추천 모니터링 iOS/Android 패널 제외
- `APPINFO_TBL` 기반 추천 모니터링 계산 로직 제외
- 앱 운영정보 화면 제외

다만 공수의 본체는 여전히 `업무보고`, `프로젝트/페이지/트래킹`, `저장 계약`, `검색/다운로드`, `통계`, `권한/검증`에 있으므로, 총 공수가 크게 줄어들지는 않는다고 보는 것이 맞습니다.

### 8.2 현재 범위 기준 예상 공수

| 구분 | 범위 | 개발 공수 | QA/안정화 | 총 공수 |
| --- | --- | --- | --- | --- |
| 1차 오픈 | 로그인, 업무보고, 프로젝트/페이지, 트래킹, 개인 검색/다운로드, 대시보드(추천 모니터링 제외), QA/모니터링 통계, 비밀번호 변경 | 17~23 MD | 5~6 MD | 22~29 MD |
| 2차 관리자 툴 포함 시 | 1차 오픈 + 별도 관리자 툴(전체 업무보고, Type, Service Group, 멤버 관리) | 20~28 MD | 6~7 MD | 26~35 MD |

해석:

- 기존 A 시나리오 `23~31 MD`에서 추천 모니터링 제외를 반영해 `22~29 MD` 수준으로 조정합니다.
- 기존 B 시나리오 `28~38 MD`에서 앱 운영정보와 추천 모니터링 연계 제외를 반영하면, 별도 관리자 툴까지 포함한 참고치는 `26~35 MD` 수준입니다.
- 불명확 레거시 꼬리까지 다시 포함하면 공수는 다시 빠르게 증가할 수 있습니다.

### 8.3 단계별 예상 공수

| 단계 | 추정 공수 | 포함 작업 |
| --- | --- | --- |
| Phase 0. 착수 및 기준선 확정 | 1~2 MD | 범위 확정, 메뉴 매핑, 제외 항목 정리, 완료 기준 정의 |
| Phase 1. Supabase 기반 구축 | 2~3 MD | Auth, `members`, 초기 적재 구조, 계정 바인딩, 최소 RLS |
| Phase 2. 저장 계약과 서버 함수 구현 | 4~6 MD | 업무보고/프로젝트/페이지/트래킹 저장 규칙, null 방지, RPC/Edge Functions |
| Phase 3. 1차 화면 구현 | 10~12 MD | 업무보고, 프로젝트/페이지, 트래킹, 개인 검색/다운로드, 대시보드(추천 모니터링 제외), 통계 |
| Phase 4. 통합 검증과 오픈 준비 | 5~6 MD | 회귀 테스트, 데이터 검증, 다운로드 검증, 오픈 점검 |
| Phase 5. 2차 관리자 툴 구현 | 4~6 MD | 전체 업무보고, Type, Service Group, 멤버 관리의 별도 관리자 툴 구현 |

즉, 현재 합의된 범위 기준으로는 `Phase 0~4 = 22~29 MD`가 1차 목표이고, `Phase 5`는 `2026-04-13` 이후 별도 관리자 툴로 `4~6 MD` 정도를 추가로 보는 것이 현실적입니다.

## 12. 결론

이 계획의 핵심은 `실제 사용하는 범위만 먼저 확정하고`, `저장 계약과 인증 구조를 먼저 세운 뒤`, `업무보고/프로젝트/트래킹을 병렬 구현`하는 것입니다.

즉, 본 작업은 저장소 전체를 기계적으로 옮기는 작업이 아니라, `실사용 기능만 빠르게 이관`하면서도 `보안과 저장 정합성을 복구하는 재구현 작업`으로 관리해야 합니다.

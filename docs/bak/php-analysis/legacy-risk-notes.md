# 레거시 리스크 및 해석 메모

## 1. 목적

이 문서는 현재 구조를 이해하면서 반드시 같이 봐야 하는 레거시 제약과 잠재 리스크를 정리한 메모입니다.

## 2. 핵심 리스크

### 2.1 SQL 하드코딩과 입력값 직접 결합

- 대부분의 SQL이 문자열 결합 방식입니다.
- Prepared Statement가 없습니다.
- 화면별로 유효성 검사 수준이 다릅니다.

영향:

- SQL Injection 위험
- 조회/수정 조건 누락 시 대량 데이터 오염 위험

### 2.2 데이터 중복 저장

- `TASK_TBL`에 프로젝트명, 페이지명, 서비스명, 타입2 텍스트를 모두 복제 저장합니다.
- 일부 레코드만 `task_pj_report_num`, `task_page_report_num`을 채웁니다.

영향:

- 프로젝트/페이지명 변경 시 과거 이력 불일치 가능
- 정합성 유지 로직이 DB가 아니라 PHP 파일에 흩어짐

### 2.3 날짜 포맷 혼재

확인된 포맷:

- `Y-m-d`
- `ymd`
- `ym`
- `Ymd`

영향:

- 문자열 비교에 의존한 집계/검색 로직의 취약성
- 월별 통계와 일별 입력 로직이 혼동되기 쉬움

### 2.4 HTML fragment 의존 API

- 서버 응답이 JSON이 아니라 DOM fragment입니다.
- 프런트엔드가 응답 구조를 강하게 가정합니다.

영향:

- 화면 구조 변경 시 API도 함께 깨질 가능성 높음
- 재사용성과 테스트 자동화가 낮음

### 2.5 타입2 숫자 ID 하드코딩

- [`webapp/js/category.js`](../../webapp/js/category.js)에서 `type_num` 값별로 폼 필드를 다르게 생성합니다.
- 예: 특정 숫자는 페이지 선택형, 특정 숫자는 URL만, 특정 숫자는 시간 480 고정 등.

영향:

- 업무 분류 기준을 바꾸기 어렵고, DB 마스터와 프런트엔드 코드가 동시 수정돼야 함
- 타입 관리 화면에서 단순 CRUD만으로는 실제 입력 규칙이 바뀌지 않음

### 2.6 숨김 화면과 호출처 불명 코드

확인된 항목:

- 메뉴 미노출 화면: `allreport.php`, `type.php`, `service_group.php`, `members.php`, `new_member.php`
- 호출처 불명 엔드포인트: `manager_*`
- 클라이언트는 `noti.php` 실시간 스트림을 기대하지만 저장소에는 파일이 없음
- SQL 덤프에는 `AGITNOTI_TBL`, `AGITNOTI_OPT_TBL`이 있으나 현재 코드에서는 `NOTI_TBL`만 참조함

영향:

- 운영 기능 누락 위험
- 실제 사용 여부 확인 없이 삭제하면 운영 사고 가능
- 알림 도메인에서 코드와 DB 구조가 이미 분기되었을 가능성

## 3. 코드상 구체 이슈 메모

### 3.1 DB 접속 정보 하드코딩

- [`webapp/dbcon/connect.php`](../../webapp/dbcon/connect.php)에 DB 호스트, 계정, 비밀번호가 직접 들어 있습니다.

### 3.2 세션/권한 체크 일관성 부족

- 일부 파일만 `user_level`을 엄격하게 확인합니다.
- 메뉴 미노출 화면의 직접 접근 차단 여부는 파일마다 다릅니다.

### 3.3 응답 타깃 누락 흔적

- `report.js`는 `#alloutput` 재로딩 코드를 포함하지만 현재 `report.php`에서 해당 DOM이 확인되지 않습니다.
- `appinfo.js`는 `#script_empty`를 응답 타깃으로 사용하지만 현재 `appinfo.php`에 해당 DOM이 없습니다.

### 3.4 중복/오타성 호출 흔적

- `pjname_pjpage()`는 `pj_sv_tran.php`를 두 번 호출하고, 두 번째 결과를 `#sv_group`에 넣습니다.
- 의도상 `pj_sv_g_tran.php`와 중복/오타 가능성이 큽니다.

### 3.5 세션 타임아웃 로직 미완성

- `login/sessioncount.php`는 존재하지만 현재 로그인 흐름에서 의미 있게 동작한다고 보기 어렵습니다.

### 3.6 APPINFO 기준정보의 부분 생존

- `pages/appinfo.php`와 `dbcon/list_appinfo*.php` 계열이 남아 있어 메뉴에서 직접 접근 가능합니다.
- `dbcon/recommandmoni.php`가 `APPINFO_TBL`을 읽어 Dashboard 추천 모니터링 목록을 계산합니다.
- 반면 업무보고, 프로젝트, 통계의 핵심 저장 구조와 직접 연결되지는 않습니다.
- SQL 덤프 기준 데이터 수정일이 대부분 오래되어 현재 운영의 1차 마스터라기보다 과거 스펙 잔존 데이터일 가능성이 큽니다.

영향:

- `APPINFO_TBL`은 완전 미사용으로 단정하면 안 되지만, 핵심 운영 테이블로 과대평가해도 안 됩니다.
- 삭제 후보 검토 시에는 "대시보드 추천 모니터링 대체 여부"와 "앱 운영정보 화면 실사용 여부"를 먼저 확인해야 합니다.

## 4. 유지보수 시 해석 원칙

1. 프로젝트명/페이지명 변경은 화면 표시 수정이 아니라 과거 이력 오염 가능성 문제로 봐야 합니다.
2. 타입 추가/수정은 DB 마스터만 보지 말고 `category.js`의 숫자 분기까지 같이 확인해야 합니다.
3. 화면 누락처럼 보여도 숨김 관리 화면일 수 있으므로 메뉴 미노출 파일을 먼저 버리지 않아야 합니다.
4. 통계 화면 수치 이상은 차트 코드보다 날짜 문자열 규칙과 집계 SQL부터 의심해야 합니다.

## 5. 이해를 돕는 결론

이 시스템의 복잡도는 화면 수보다 "텍스트 복제 데이터", "하드코딩된 타입 분기", "부분 갱신 DOM 계약"에서 나옵니다. 따라서 기능 파악 시 파일 개수보다 다음 축을 우선 보는 것이 정확합니다.

- `TASK_TBL`에 무엇이 어떻게 저장되는가
- `PJ_TBL` / `PJ_PAGE_TBL` 변경이 어디까지 전파되는가
- `TYPE_TBL`과 `category.js`가 어떻게 결합되어 있는가
- `pages/*.php`와 `dbcon/*.php`가 어느 DOM을 교체하는가

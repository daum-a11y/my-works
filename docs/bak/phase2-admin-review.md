- 기준일: 2026-03-24

## 2. 기존 transition 문서 기준 현재 2차 작업 해석

기존 문서 기준 2차 작업은 아래 성격으로 정의돼 있습니다.

- 1차 사용자 화면 오픈 이후
- 별도 관리자 툴 또는 별도 관리자 영역으로 분리
- 기본 포함 후보:
  - 전체 업무보고 화면
  - Type 관리
  - Service Group 관리
  - 멤버 관리/신규 멤버 생성

이 해석은 `php-operation_tool` 기준으로는 자연스럽지만, 이번 세션에서 확인한 후속 운영툴 `linkagelab-a11y-workmanage`의 실제 상태와는 일부 차이가 있습니다.

## 3. 이번 세션에서 확인한 핵심 사실

### 3.1 현재 관리자성 기능의 실제 중심축

현재 코드에서 관리자성 기능의 실질 중심은 아래입니다.

- 전체 업무 검색
- 검색 결과 행 단위 수정/삭제
- 엑셀 다운로드
- 사용자 관리
- Type 유효성검증과 bulk 치환
- Service Group 유효성검증과 bulk 치환
- 리소스 집계 조회

즉, 현재 운영툴의 관리자 기능은 `마스터 신규 생성`보다 `오염된 업무데이터 조회/정리/집계`에 더 가깝습니다.

### 3.2 Type / Service Group 관리모드의 현재 상태

이번 세션 판단 기준으로 아래 두 기능은 스펙아웃입니다.

- 업무타입 관리모드 CRUD
- 서비스 그룹 관리모드 CRUD

코드에는 남아 있지만, 2차 개발의 기본 범위로 복원할 대상으로 보지 않는 것이 맞습니다.

### 3.3 아지트 QA알리미 상태

아래 기능도 코드상 잔존하지만 현재 스펙아웃입니다.

- 아지트 QA알리미 로그/옵션 화면
- 관련 API와 `AGITNOTI_*` 연동

따라서 2차 관리자 작업 범위에서 제외해야 합니다.

### 3.4 프로젝트 영역 상태

프로젝트 관련 코드는 다음처럼 반쯤 남아 있습니다.

- `Project.vue`와 `components/project/*` 존재
- 프로젝트 생성/목록/삭제 API 존재
- 하지만 라우터에서는 비활성

따라서 현재 시점에서 프로젝트 관리 화면을 2차 기본 범위로 포함하면 안 됩니다.

다만 `PJ_TBL` 자체는 아직 개인 업무보고 화면의 `프로젝트 검색` 보조 데이터로 사용됩니다. 즉, 프로젝트 도메인이 완전히 사라진 것은 아니지만, `관리 화면 복원`과 `입력 보조 검색 유지`는 구분해서 봐야 합니다.

### 3.5 데이터 품질 관점의 가장 중요한 사실

현재 관리자 기능에서 가장 중요한 문제는 `TASK_TBL`의 문자열 데이터 품질입니다.

구체적으로:

- `task_type1`, `task_type2`
- `task_svc_group`, `task_svc_name`

이 값들이 마스터와 어긋날 수 있기 때문에, 현재 관리자 화면의 핵심 기능은 CRUD보다 `validation + bulk correction`입니다.

## 4. 2차 작업 범위에 대한 검토 결론

### 4.1 기존 2차 범위 중 유지해야 할 항목

아래 항목은 2차 작업의 핵심 범위로 유지하는 것이 맞습니다.

1. 전체 업무보고 화면
2. 멤버 관리/신규 멤버 생성

다만 여기서 `전체 업무보고 화면`은 PHP의 `allreport.php`를 기계적으로 옮긴다는 의미보다, 현재 운영툴 기준의 아래 기능 묶음으로 재해석하는 것이 맞습니다.

- 전체 검색
- 필터링
- 결과 목록 조회
- 행 단위 수정/삭제
- 엑셀 다운로드

### 4.2 기존 2차 범위 중 재해석이 필요한 항목

기존 문서의 아래 항목은 이름 그대로 구현하는 대신 목적을 바꿔 해석해야 합니다.

1. Type 관리
2. Service Group 관리

권장 재해석:

- `Type 관리` -> `Type 유효성검증 및 업무데이터 보정`
- `Service Group 관리` -> `Service Group 유효성검증 및 업무데이터 보정`

즉, 2차 관리자 기능의 핵심은 마스터 CRUD가 아니라 아래입니다.

- 현재 마스터 목록 조회
- orphan 데이터 검출
- 대상 업무 row 조회
- 일괄 치환

### 4.3 기존 2차 범위에서 제외해야 할 항목

아래 항목은 현재 세션 기준 2차 기본 범위에서 제외하는 것이 맞습니다.

1. 업무타입 관리모드 CRUD
2. 서비스 그룹 관리모드 CRUD
3. 아지트 QA알리미
4. 비활성 프로젝트 관리 화면 복원

이 항목들은 코드에 잔존하지만, 현행 운영 기준의 우선순위 기능으로 보기 어렵습니다.

## 5. 2차 작업의 권장 범위 재정의

### 5.1 필수 범위

2차 관리자 작업의 필수 범위는 아래처럼 재정의하는 것이 적절합니다.

| 영역 | 포함 기능 | 비고 |
| --- | --- | --- |
| 관리자 검색 | 기간/사용자/type/svc 조건 검색, 목록 조회, 인라인 수정/삭제, 엑셀 다운로드 | 기존 `전체 업무보고`의 현실적 대응 |
| 사용자 관리 | 목록 조회, 생성, 수정, 활성/비활성, 비밀번호 초기화 | 유지 필요 |
| Type validation | orphan type 검출, 대상 row 조회, 일괄 치환 | CRUD 대신 우선 구현 |
| Service validation | orphan svc 검출, 대상 row 조회, 일괄 치환 | CRUD 대신 우선 구현 |

### 5.2 조건부 범위

아래는 운영 필요성이 확인되면 2차에 포함할 수 있습니다.

| 영역 | 포함 기능 | 판단 기준 |
| --- | --- | --- |
| 리소스 집계 | 일간 작성현황, 월간 MM, 타입/서비스 요약 | 운영에서 관리자 집계 화면이 실제 필요한지 확인 필요 |
| 프로젝트 참조 | 개인 업무입력용 프로젝트 검색 API/lookup | 1차 업무보고 폼이 프로젝트 기반 입력을 유지하는 경우 필요 |

### 5.3 제외 범위

| 영역 | 제외 기능 | 제외 사유 |
| --- | --- | --- |
| Type master CRUD | 신규/수정/삭제 관리모드 | 현재 스펙아웃 |
| Service master CRUD | 신규/수정/삭제 관리모드 | 현재 스펙아웃 |
| 아지트 알리미 | 로그/옵션 화면 및 연계 | 현재 스펙아웃 |
| 프로젝트 관리 UI | 프로젝트 목록/생성/삭제 화면 | 현재 비활성, 우선순위 낮음 |

## 6. 기술적 설계 시사점

### 6.1 Type / Service CRUD를 제외하더라도 마스터 데이터는 필요함

관리모드를 스펙아웃하더라도 아래는 여전히 필요합니다.

- validation 대상 비교 기준
- 개인 업무보고 입력폼의 선택 옵션
- 관리자 검색 필터 옵션

따라서 2차 개발에서는 `master CRUD를 앱 안에 둘지`와 별개로, 최소한 `read-only master source`와 `운영 반영 절차`를 별도로 정해야 합니다.

### 6.2 2차 관리자 기능의 핵심은 데이터 보정 도구임

현재 구조상 2차 관리자 기능은 일반적인 `백오피스 마스터 관리`가 아니라 아래에 더 가깝습니다.

- 레거시 문자열 데이터 정리 도구
- 운영 중 잘못 적재된 업무 이력 보정 도구
- 대량 조회/내려받기 도구

이 해석이 맞지 않으면 우선순위가 쉽게 뒤집힙니다.

### 6.3 프로젝트 기능은 입력 보조와 관리 화면을 분리해서 판단해야 함

현재 `PJ_TBL`은 완전 제거 대상이 아닙니다. 하지만 그 이유는 `프로젝트 관리 화면`이 살아 있어서가 아니라, `개인 업무보고 폼의 프로젝트 검색`이 아직 의존하기 때문입니다.

따라서 2차 개발에서는 아래를 분리해야 합니다.

- 프로젝트 검색 lookup 유지 여부
- 프로젝트 관리 UI 복원 여부

두 항목을 하나로 묶으면 범위가 불필요하게 커집니다.

## 8. 최종 결론

이번 세션 기준으로 2차 작업은 `관리자용 마스터 CRUD 복원`이 아니라 `관리자용 조회/보정/운영 도구 정리`로 보는 것이 맞습니다.

정리하면:

1. `전체 업무보고`는 `전체 검색 + 수정/삭제 + 엑셀`로 재해석해야 합니다.
2. `Type 관리`, `Service Group 관리`는 `CRUD`보다 `validation + bulk correction`이 우선입니다.
3. `멤버 관리`는 그대로 유지 대상입니다.
4. `업무타입 관리모드`, `서비스 그룹 관리모드`, `아지트 QA알리미`, `비활성 프로젝트 관리 UI`는 현재 2차 범위에서 제외하는 것이 타당합니다.

이 방향으로 정리해야 `./docs/transition`의 2차 개발 범위가 실제 운영 근거와 맞고, 구현 우선순위도 흔들리지 않습니다.

# UI 정책

## 화면 대응 범위

- 현재 서비스는 **데스크톱 전용(Desktop Only)**으로 운영됩니다.
- 모바일/태블릿 반응형 별도 레이아웃은 현재 지원하지 않습니다.
- 반응형 대응이 필요할 경우 `레이아웃 정책`과 `전역/공통 스타일`을 함께 정비하는 별도 작업이 필요합니다.

## 공통 버튼 규칙

- 버튼 레이블은 상태에 따라 바꾸지 않습니다.
- 버튼 내부에 로딩 스피너, 진행 아이콘, `...중` 문구를 넣지 않습니다.
- 진행 상태 제어가 필요하면 버튼 비활성화나 버튼 외부 피드백으로 처리합니다.

## 프로젝트 관리 화면 규칙

- `프로젝트 관리` 목록 화면은 `통계` 화면의 필터 문법을 기준으로 맞춥니다.
- 필터는 `라벨 + 입력`, 우측 끝 `검색/초기화` 액션 조합을 기본으로 하며, 버튼 높이와 정렬은 `src/features/stats/shared.module.css` 기준을 따릅니다.
- 목록 표는 `업무내역 조회`, `통계` 표와 같은 밀도와 헤더 문법을 사용합니다.
- 표 셀에서 의미 없는 강조 표현을 금지합니다. 플랫폼 배지, 프로젝트명 볼드, 전체 URL 노출 같은 일회성 표현을 넣지 않습니다.
- 링크가 필요한 셀은 전체 URL 문자열 대신 `링크` 같은 짧은 액션 라벨로 노출합니다.
- 목록 화면은 `조회` 역할만 가집니다. `추가`, `수정`은 별도 페이지에서 처리합니다.
- 프로젝트 화면 전용 버튼 색을 만들지 않습니다. 주 버튼, 보조 버튼, 위험 버튼은 기존 공용 역할색을 그대로 사용합니다.

## 프로젝트 추가/수정 화면 규칙

- `프로젝트 추가`, `프로젝트 정보 수정`은 같은 폼 문법을 사용합니다.
- 헤더는 공용 페이지 제목 행만 사용합니다. 별도 hero, 소개 문구, 과한 상단 여백을 금지합니다.
- 프로젝트 기본 정보 폼은 `한 행 = 라벨 + 입력칸` 문법으로 고정합니다. 필드를 가로 카드처럼 나열하거나 화면마다 다른 폼 배치를 만들지 않습니다.
- 라벨은 좌측 고정폭, 입력칸은 우측 고정폭으로 맞추고, 긴 필드도 같은 행 문법 안에서 처리합니다.
- `프로젝트 추가` 화면의 하단 액션은 오른쪽에 `취소`, `저장하기`만 둡니다.
- `프로젝트 정보 수정` 화면의 하단 액션은 왼쪽에 `삭제`, 오른쪽에 `취소`, `저장하기`를 둡니다. `저장하기`는 항상 가장 오른쪽에 둡니다.
- `목록으로`와 `취소`처럼 의미가 겹치는 액션을 같은 화면에 중복 배치하지 않습니다.
- `페이지 목록`는 프로젝트 기본 정보 폼과 같은 패널 안에 가두지 않습니다. 수정 화면에서는 별도 패널로 분리합니다.
- 버튼 위계는 `저장하기 = 주 액션`, `취소 = 보조 액션`, `삭제 = 위험 액션`으로 고정합니다.
- 리포터, 리뷰어 같은 멤버 선택 목록은 `legacyUserId(한글 이름)` 형식으로 노출합니다.

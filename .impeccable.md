## Design Context

### Core Concept
`Precision Ops Console`

이 제품은 접근성팀과 운영 담당자가 하루 종일 사용하는 `고대비·고밀도 운영 콘솔`이다.  
예쁜 서비스형 화면이 아니라, 빠르게 읽고 바로 수정하고 상태를 즉시 판단하는 `상용 백오피스형 업무 도구`로 보여야 한다.

### Users
접근성팀과 내부 운영 담당자가 업무시간 동안 데스크톱과 노트북에서 장시간 사용하는 운영관리툴이다. 핵심 작업은 업무보고 기록, 프로젝트와 페이지 관리, 모니터링 및 QA 상태 확인, 관리자 검색과 수정이다. 사용자는 화면을 감상하지 않고 빠르게 읽고 입력하고 수정해야 한다.

### Usage Context
- 기본 사용 환경은 데스크톱과 노트북이다. 긴 집중 업무, 키보드 사용, 다량의 표와 폼 조작을 전제로 한다.
- 태블릿은 현장 확인과 간단 수정까지 허용하는 보조 환경이다.
- 모바일은 조회와 빠른 확인까지 우선 지원하되, 핵심 기능을 숨기지 않는다.
- 저시력 사용자가 실제 사용자군에 포함되므로, 판독성과 대비가 미학보다 우선이다.

### Performance & Motion Constraints
- 이 제품은 빠르게 느껴져야 한다. 장식용 애니메이션보다 반응 속도와 상태 피드백이 우선이다.
- 애니메이션은 `transform`과 `opacity` 중심으로 제한하고, 레이아웃 속성 애니메이션은 금지한다.
- `prefers-reduced-motion`를 반드시 존중한다.
- 페이지 전환, 메뉴 활성화, 필터 반응, 저장 성공/실패 피드백처럼 의미가 분명한 곳에만 모션을 사용한다.

### Audience Copy Context
- 사용자는 비기술 직군이 아니더라도 개발자처럼 시스템 용어를 당연히 이해한다고 가정하면 안 된다.
- 사용 시점의 감정 상태는 대체로 급하고 피곤하다. 읽고 해석하는 데 시간이 걸리는 문구는 실패다.
- 문구는 업무 지시문처럼 짧고 단정해야 하며, 라벨과 버튼은 같은 뜻을 반복 없이 바로 이해되어야 한다.

### Brand Personality
정확함, 단정함, 신뢰감. 소비자용 서비스처럼 친절하거나 화려한 인상보다, 상용 B2B 운영 콘솔처럼 조용하고 단단한 인상이 우선이다.

### Frontend Design Direction
- **Purpose**: 업무보고, QA, 모니터링, 관리자 검색과 수정처럼 반복적이고 데이터 중심인 일을 가장 빠르게 처리하게 만드는 운영 콘솔이다.
- **Tone**: `industrial utilitarian light console`. 장식보다 구조, 친절함보다 정확성, 분위기보다 판독 속도를 우선한다.
- **Constraints**: 저시력 사용자를 고려한 고대비, 긴 업무시간에 버티는 피로도, 데스크톱 우선, 모바일에서도 핵심 기능 유지, 과한 애니메이션 금지.
- **Differentiation**: 이 제품이 기억되어야 하는 지점은 “예쁜 박스”가 아니라 `단호한 정보 위계`, `빠른 스캔`, `분명한 상태`, `얕은 패널 깊이`다.
- **AI Slop Rejection**: 동일 카드 그리드, 서비스형 hero, 그라데이션 임팩트, 과한 그림자, 예쁜 pill 위젯, 설명형 소개 문구를 사용하지 않는다.

### Signature Visual Language
- 좌측은 `탐색 레일`, 중앙은 `작업판`, 상단은 `컨텍스트 바`로 읽혀야 한다. 화면을 박스로 쪼개는 대신, 구조 축이 먼저 보여야 한다.
- 첫 인상은 “단단하고 빠른 콘솔”이어야 한다. `친절한 SaaS`, `예쁜 대시보드`, `부드러운 협업툴`처럼 보이면 실패다.
- 강한 지점은 세 곳만 허용한다: `현재 위치`, `핵심 상태`, `주 액션`. 나머지는 분명히 약해야 한다.
- 기억에 남아야 하는 것은 색 효과가 아니라 `정확한 정렬`, `짧은 헤더`, `좁은 탐색`, `넓은 데이터 면적`이다.
- 이 제품의 시그니처는 `고밀도이지만 숨 막히지 않는 정렬감`이다. 복잡한 업무를 차갑고 빠르게 읽히게 만드는 구조 자체가 인상이어야 한다.

### Frontend Anti-Pattern Ban
- 동일한 크기의 카드 그리드 반복을 금지한다.
- 서비스형 hero, 소개성 카피, decorative metric block을 금지한다.
- “박스 안의 박스 안의 박스” 구조를 금지한다.
- 연한 회색선으로 모든 경계를 해결하는 방식을 금지한다.
- 중앙 정렬 남용, 넓은 사이드바, 두꺼운 헤더, 과한 상하 여백을 금지한다.
- 예쁜 pill 버튼, 떠 있는 위젯 카드, 의미 없는 라운드 박스를 금지한다.
- 퍼플/블루 그라데이션, 네온 포인트, glassmorphism, gradient text를 금지한다.
- 설명이 제목을 다시 반복하는 헤더 구조를 금지한다.

### Aesthetic Direction
라이트 모드 중심의 상용 운영 콘솔. 시각 기준은 `표와 필터 중심`, `중립 배경`, `작은 반경`, `적은 색`, `적은 장식`이다.  
메인 톤은 차갑고 중립적인 회백/슬레이트 계열로 유지하되, 너무 무기력한 회색 화면으로 끝나면 안 된다. 색 시스템은 `중립 80 / 강조 15 / 상태 5` 비율로 운용한다. 현재 프로젝트의 적합한 강조색 축은 청록이 아니라 인디고/슬레이트 계열이다.  
패널은 배경 톤 차이와 간격으로 구분하고, 보더는 구조적으로 꼭 필요한 곳에만 제한한다. 다만 저시력 사용자를 위해 기능 구분용 대비는 약화하면 안 된다. 로그인 화면도 SaaS 랜딩 카드처럼 보이면 안 되며, 앱 전체와 같은 제품 문법 안에서 단순하고 조밀하게 보여야 한다.  
금지 방향은 다음과 같다: 서비스형 hero, 과한 빈 여백, 연한 보더 남발, 화면마다 다른 버튼/패널 문법, 따뜻한 종이톤과 차가운 포인트색의 혼합, 과한 pill 버튼과 위젯형 카드. 추가로 금지한다: 회색만 잔뜩 깔린 무채색 내부툴, 의미 없는 파스텔 배경, 페이지마다 다른 강조색 사용.

### Design Principles
1. 모든 화면은 `제목 -> 필터/상태 -> 표/폼` 순으로 읽혀야 하며, 설명형 hero나 소개 문구는 최소화한다.
2. 가독성이 미학보다 우선이다. 명도대비 기준은 주요 콘텐츠 7:1 이상, 서브 콘텐츠 4.5:1 이상이다. 본문, 제목, 라벨, 표 헤더, 입력값, 버튼 텍스트는 주요 콘텐츠 기준으로 보고, 메타 텍스트, 보조 설명, 보조 액션은 서브 콘텐츠 기준 아래로 떨어지면 안 된다.
3. 그룹 구분은 보더보다 표면 톤과 여백으로 해결하되, 기능 구분에 필요한 대비는 분명해야 한다. 입력, 표 래퍼, 구조 경계, 선택 상태, 포커스 링은 배경과 확실히 분리되어야 하며 흐린 회색선으로 뭉개면 안 된다.
4. 컴포넌트 문법은 전 화면에서 같아야 한다. 버튼, 패널, 탭, 배지, 필터 바, 표는 같은 제품처럼 보여야 하며, 로그인도 예외가 아니다.
5. 색은 의미 전달과 위계 형성에만 사용한다. 기본 화면은 슬레이트 계열 중립색으로 유지하고, 강조색은 액션과 선택 상태, 현재 위치, 핵심 링크, 차트의 주 시리즈에만 사용한다. 상태색은 성공/주의/위험에만 제한한다. placeholder, 비활성 버튼, 보조 라벨도 읽힘을 해치지 않는 대비를 유지해야 한다.
6. 타이포는 고정 스케일만 사용한다. 페이지 제목 `1.25rem`, 섹션 제목 `1.125rem`, 본문 `1rem`, 라벨 `0.9375rem`, 메타 `0.875rem`, 표 헤더/상태 캡션 `0.8125rem`을 기준으로 하고, 임의의 `0.82`, `0.84`, `0.92`, `1.02` 같은 중간값을 새로 만들지 않는다.
7. 색 역할은 고정한다. 인디고 계열은 `현재 위치 / 주 액션 / 선택 / 강조 링크`, 그린은 `완료 / 정상`, 앰버는 `주의 / 마감 임박`, 레드는 `오류 / 위험`, 슬레이트는 `비활성 / 보조 상태`에만 사용한다. 같은 의미에 다른 색을 쓰지 않는다.
8. 강조색은 화면 전체를 칠하는 용도가 아니다. 메뉴 활성선, 선택 행, 주요 버튼, 핵심 수치 한두 곳에만 쓰고, 패널 배경 전체를 연한 강조색으로 물들이지 않는다.
9. 차트와 상태 배지는 더 선명해야 하지만 더 많아지면 안 된다. 차트는 `주 시리즈 1색 + 비교군 중립색`, 배지는 `색 + 텍스트`를 함께 사용하고 색만으로 의미를 전달하지 않는다.
10. 레이아웃은 크롬보다 데이터가 우선이다. 데스크톱 기준 사이드바는 `11.5rem ~ 12.5rem`, 헤더는 `56px ~ 60px` 수준으로 억제하고, 본문은 첫 화면에서 핵심 데이터가 바로 보여야 한다.
11. 화면 구조는 `좌측 네비게이션 / 상단 컨텍스트 바 / 본문 작업영역` 3단만 유지한다. 본문 안에서 또 hero, 또 summary card, 또 panel을 중첩하지 않는다.
12. 패널 깊이는 최대 2단계까지만 허용한다. `페이지 배경 -> 작업 패널/표 래퍼`까지만 쓰고, 패널 안에 또 패널을 넣지 않는다.
13. 대시보드와 현황 화면은 동일 카드 반복으로 풀지 않는다. `31개 날짜 카드`, `동일 메트릭 카드 n개`, `같은 크기 박스 배열`은 금지하고, 매트릭스·표·리스트·압축 지표 중심으로 정리한다.
14. 간격은 리듬이 있어야 한다. 같은 그룹 내부는 `8px ~ 12px`, 섹션 내부는 `16px ~ 24px`, 독립 섹션 사이는 `24px ~ 32px`를 기본으로 하고 모든 영역을 같은 패딩으로 두르지 않는다.
15. 메뉴는 위젯이 아니라 탐색 구조여야 한다. 제품명, 그룹 레이블, 링크, 현재 위치가 한 번에 읽혀야 하며, 그룹 제목과 링크를 같은 크기와 무게로 두지 않는다.
16. 모바일과 태블릿에서도 정보 구조는 바꾸지 않는다. 단순히 줄 세우는 것이 아니라, 사이드바는 접고 본문 작업 흐름은 유지하되 핵심 필터와 표 조작은 그대로 살아 있어야 한다.
17. 한 화면에 하나의 초점만 둔다. 현재 작업, 위험 상태, 핵심 액션 중 무엇이 제일 중요한지 먼저 정하고, 나머지는 그보다 분명히 약하게 배치한다.
18. 대시보드는 “소개 화면”이 아니라 “업무 상황판”이어야 한다. 첫 화면에 보이는 것은 실제 업무 현황, 누락, 진행 상태여야 하며 장식용 메트릭과 반복 카드가 아니어야 한다.
19. 폼과 표는 장식 요소가 아니라 핵심 인터랙션이다. 입력창, 표 헤더, 선택 행, 필터 토글은 가장 공들여 보여야 한다.
20. 시각적 인상은 “2026 상용 운영 콘솔”이어야 한다. 오래된 사내 웹앱처럼 보이는 넓은 사이드바, 둔한 헤더, 박스 반복, 연한 회색선 나열을 피한다.

### Adapt Guide
- 데스크톱이 기본 문맥이다. `1024px+`에서는 사이드바 고정, 상단 컨텍스트 바 유지, 다열 데이터 구조를 적극 사용한다.
- 태블릿 `768px~1023px`에서는 사이드바를 접을 수 있는 드로어로 바꾸고, 본문은 1열 중심으로 재배치하되 필터와 표는 같은 페이지 안에서 유지한다.
- 모바일 `320px~767px`에서는 조회와 즉시 조작 중심으로 압축한다. 필터는 접힘 구조를 허용하지만 결과 표와 핵심 액션은 숨기지 않는다.
- 터치 환경에서도 핵심 액션과 탭, 필터 토글은 최소 `44x44px`를 보장한다.
- 반응형은 “작게 줄이기”가 아니라 “업무 흐름 유지”가 목적이다. 모바일에서 다른 정보 구조를 만들지 않는다.

### Arrange Guide
- 사이드바, 헤더, 본문 3단 구조만 유지한다. 여기에 hero, intro, summary card 층을 추가하지 않는다.
- 본문 첫 스크린에서는 핵심 데이터가 보여야 한다. 제목 아래 긴 설명문과 박스 반복으로 첫 화면을 소비하지 않는다.
- 같은 크기의 카드 그리드 반복을 금지한다. 상태 반복은 표, 리스트, 매트릭스, 압축 지표로 바꾼다.
- 패널은 많은 것이 아니라 적절히 깊이가 얕아야 한다. “박스 안의 박스”를 만들지 않는다.
- 공간 리듬은 의도적으로 준다. 관련 항목은 촘촘하게, 다른 작업 단위는 넓게 벌린다.

### Bolder Guide
- 이 제품의 bold는 화려함이 아니라 `더 큰 위계 차`, `더 분명한 선택 상태`, `더 단호한 구조`다.
- 크게 보여야 하는 것은 페이지 제목이 아니라 현재 작업, 주요 상태, 선택 행, 경고 액션이다.
- 극적인 대비는 허용하되, 다크모드 네온, 그라데이션 텍스트, 과장된 그림자 같은 AI식 효과는 금지한다.
- 한 화면에 하나의 초점만 둔다. 모든 패널과 메뉴를 동시에 강조하지 않는다.
- bold를 주는 방식은 크기·무게·정렬·여백 차이여야 한다. 장식 효과로 시선을 끄는 방식은 금지한다.

### Colorize Guide
- 기본 톤은 슬레이트 중립색으로 유지하되, 현재 위치와 핵심 액션은 한눈에 보이게 인디고 축을 분명히 사용한다.
- 상태색은 성공/주의/위험/정보 외에는 늘리지 않는다.
- 패널 배경을 색으로 칠하지 않는다. 색은 버튼, 활성 메뉴, 선택 행, 링크, 상태 배지처럼 의미가 있는 위치에만 쓴다.
- 회색만 잔뜩 남는 무채색 화면도 금지한다. 구조 위계와 탐색을 돕는 최소한의 역할색은 반드시 사용한다.

### Animate Guide
- 모션은 빠르고 짧아야 한다. 피드백 `100~150ms`, 상태 전환 `200~250ms`, 패널/드로어 진입 `250~320ms` 안에서 해결한다.
- 허용 모션: 메뉴 활성화 전환, 탭/필터 전환, 패널 확장/축소, 저장 성공/실패 피드백, 라우트 진입 시 짧은 페이드.
- 금지 모션: 상시 떠다니는 장식, 과한 스태거, 바운스/엘라스틱, 레이아웃 재배치 애니메이션, 반복되는 주의 끌기 효과.
- reduced motion에서는 장식 모션을 제거하고 상태 변화만 즉시 반영한다.

### Clarify Guide
- 메뉴와 제목은 사용자가 실제 업무에서 쓰는 용어를 그대로 쓴다. `Dashboard` 같은 포괄 제목보다 기능명이 우선이다.
- 버튼은 `저장`, `등록`, `로그아웃`처럼 행동을 직접 말해야 한다. `확인`, `완료`, `실행` 같은 모호한 표현을 줄인다.
- 빈 상태는 “없습니다”로 끝내지 않는다. 왜 비어 있는지, 무엇을 하면 되는지 같이 보여준다.
- 오류 문구는 사용자를 탓하지 말고, 원인과 다음 행동을 같이 말한다.
- 같은 개념에는 같은 단어만 쓴다. `보고서`, `리포트`, `업무보고`를 같은 의미로 섞어 쓰지 않는다.

### Typeset Guide
- 본문은 `1rem` 아래로 내리지 않는다. 장시간 사용하는 운영툴이므로 가독성 하한을 지킨다.
- 제목은 `700~800`, 라벨은 `600~700`, 본문은 `400~500`, 메타와 표 헤더는 `600~700`으로 역할을 고정한다. 같은 역할에 다른 무게를 섞지 않는다.
- 페이지 제목과 섹션 제목은 자간을 약간 조이고, 표 헤더와 그룹 레이블은 자간을 약간 넓힌다. 본문과 라벨은 과한 자간 조작을 하지 않는다.
- 제목 줄높이는 `1.1~1.2`, 본문과 설명은 `1.45~1.6`, 메타는 `1.35~1.45`로 본다.
- 숫자와 날짜, 표 수치는 반드시 `tabular-nums`를 우선 사용한다.
- 메뉴, 탭, 표 헤더처럼 반복 스캔하는 요소는 “예쁜 제목체”가 아니라 빠르게 읽히는 몸체 문법을 쓴다.
- 한 화면 안에서 같은 역할의 텍스트가 크기, 무게, 자간이 다르면 실패다.
- 전역 타이포 토큰은 역할 기준으로만 둔다: `page-title 1.25rem`, `section-title 1.125rem`, `body 1rem`, `label 0.9375rem`, `meta 0.875rem`, `table 0.8125rem`, `badge 0.75rem`. 화면별 중간 크기를 새로 만들지 않는다.
- 전역 줄높이 토큰도 고정한다: `title 1.2`, `label 1.3`, `body 1.5`, `meta 1.4`, `tight 1.15`.
- 전역 굵기 토큰은 `400 / 500 / 600 / 700 / 800`만 사용한다. `560`, `650` 같은 임의 weight를 만들지 않는다.
- 입력값은 기본적으로 `body`, 라벨은 `label`, 보조 액션은 `meta`, 메인 버튼 텍스트는 `label`, 표 헤더와 상태 캡션은 `table`만 사용한다.
- 폼에서는 예외를 둔다. 운영툴 폼의 `라벨`, `입력값`, `선택값`, `버튼 텍스트`, `보조 액션`은 모두 본문과 같은 `1rem`을 기본으로 한다. 폼 안에서만 라벨이나 버튼을 더 작게 내려 위계를 만들지 않는다.

### Form Rhythm Guide
- 폼은 `라벨 -> 컨트롤 -> 설명/오류` 3단 리듬을 공통으로 유지한다. 라벨과 컨트롤 간격은 `4px~6px`, 필드와 필드 간격은 `12px~16px`, 버튼 전 여백은 `12px` 안팎으로 맞춘다.
- 기본 입력 높이는 `40px`, 강조 입력이나 모바일 터치 상황만 `44px`까지 허용한다. 로그인처럼 단순 폼에서 `48px+` 높이는 금지한다.
- 입력 내부 패딩은 전역 토큰으로만 관리한다. 텍스트가 커 보이는 원인은 대개 폰트보다 높이와 패딩이므로, 화면별 필드 padding override를 금지한다.
- `input/select/textarea/button`에 화면별 `font-size`, `line-height`, `padding`, `min-height`를 직접 박는 방식은 예외 상황 외 금지한다. 공용 컴포넌트와 공용 토큰만 사용한다.
- 폼 안에서 제목, 라벨, 입력값, 버튼, 보조 액션의 시각 무게는 `title > label/button > input value > meta action` 순으로 읽혀야 한다.

### Audit Checklist
- 접근성: 주요 콘텐츠 7:1, 서브 콘텐츠 4.5:1, 키보드 포커스, 선택 상태, 표 헤더 판독성, 입력 경계 대비를 먼저 본다.
- 성능: 대시보드와 표 화면에서 레이아웃 애니메이션, 불필요한 재렌더, 과한 그림자/필터 효과를 금지한다.
- 테마: 하드코딩 색상과 임의 폰트 크기, 임의 radius, 임의 box-shadow를 허용하지 않는다.
- 반응형: 모바일에서 가로 스크롤, 작은 터치 타깃, 필수 기능 숨김을 우선 점검한다.
- 안티패턴: 서비스형 hero, 카드 반복, AI식 보라-파랑 그라데이션, 과한 유리질감, 데이터 없는 메트릭 카드, 설명만 많은 빈 대시보드를 금지한다.

### Harden Guide
- 긴 프로젝트명, 긴 사용자명, 긴 페이지 제목, 긴 상태 메모가 들어와도 레이아웃이 깨지지 않아야 한다. 표 셀과 메뉴, 배지, 버튼 라벨은 줄바꿈·말줄임·최소폭 규칙을 먼저 갖춰야 한다.
- 빈 데이터는 디자인 실패가 아니라 기본 상태다. 빈 표, 빈 검색 결과, 빈 대시보드, 권한 없음, 오류 상태마다 각각 다른 문구와 다음 행동이 있어야 한다.
- 입력 오류, 권한 오류, 인증 만료, 네트워크 오류, 저장 실패는 모두 화면 안에서 바로 이해되어야 한다. `오류`, `실패` 같은 추상어로 끝내지 않는다.
- 모바일과 태블릿에서 긴 텍스트, 큰 글자 확대, 다국어 확장으로도 핵심 조작이 가려지면 안 된다. 고정 폭 텍스트 상자와 잘리지 않는 버튼 라벨을 우선 확인한다.
- 날짜, 수치, 통화, 개수는 지역화와 자릿수 증가를 견뎌야 한다. 숫자는 정렬되고, 긴 숫자는 레이아웃을 밀어내지 않게 처리한다.
- 표와 폼은 `min-width: 0`, `overflow-wrap`, `text-overflow`, `line-clamp` 같은 방어 규칙을 적극 사용한다. 콘텐츠가 길다고 레이아웃이 먼저 깨지면 안 된다.
- 에러와 경고는 색만으로 전달하지 않는다. 아이콘, 제목, 설명, 다음 행동을 함께 보여준다.
- 같은 화면 안에서 일부 데이터만 실패해도 전체 화면을 막지 않는다. 부분 실패는 부분 상태로 드러내고, 나머지 작업은 계속 가능해야 한다.
- 연속 클릭, 중복 저장, 느린 응답, 새로고침 직후 상태 복원 같은 실제 운영 상황을 전제로 인터랙션을 설계한다.

### Distill Guide
- 모든 화면에서 먼저 제거할 대상은 `중복 제목`, `설명형 문단`, `의미 없는 섹션 헤더`, `반복 카드`, `장식용 보더`다.
- 한 화면의 1차 목표는 하나만 남긴다. 그 목표를 돕지 않는 카드, 메트릭, 버튼은 숨기거나 제거한다.
- 빈 상태와 요약 영역은 짧고 직접적이어야 한다. 설명을 늘리는 대신 다음 행동을 한 줄로 제시한다.
- 패널 안에서 다시 나뉘는 소제목, 보조 배경, 구분선이 많아지면 제거가 우선이다.

### Normalize Guide
- 공용 버튼, 입력, 탭, 배지, 표, 패널 문법 외의 임의 변형을 만들지 않는다.
- 같은 역할의 요소는 화면이 달라도 같은 크기, 같은 색 역할, 같은 상태 표현을 사용한다.
- 화면별 CSS에서 `input/select/textarea/button`을 직접 다시 스타일링하는 패턴을 최소화하고 공용 토큰과 공용 컴포넌트 규칙을 우선한다.
- 새 화면을 만들 때는 “예외를 추가”하는 방식이 아니라 기존 문법 안에서 조합하는 방식으로 해결한다.

### Delight Guide
- 이 제품에서 delight는 장난기보다 `정제된 반응성`이다. 빠른 저장 피드백, 분명한 선택 전환, 만족스러운 포커스 이동 정도만 허용한다.
- 성공 상태는 짧고 확실하게, 오류 상태는 거칠지 않게 보여야 한다. 축하 효과나 과한 연출은 금지한다.
- 반복 사용하는 운영툴이므로 delight는 사용성을 흐리지 않는 수준의 미세한 완성도로만 남긴다.
- 발견형 디테일은 허용하되, 핵심 업무 흐름을 방해하는 이스터에그나 장식 모션은 두지 않는다.

### Onboard Guide
- 온보딩의 목표는 기능 설명이 아니라 `첫 업무 성공`이다. 첫 보고 작성, 첫 검색, 첫 수정처럼 실제 행동으로 바로 연결되어야 한다.
- 빈 상태는 전부 온보딩 기회로 본다. “없음”으로 끝내지 말고, 이 화면에 무엇이 오고 어떻게 시작하는지 보여준다.
- 초보 사용자를 위해 설명은 짧게 두되, 숙련 사용자 작업 속도를 막는 강제 튜토리얼은 금지한다.
- 도움말은 화면 밖 문서보다 화면 안 문맥에 붙인다. 필요한 순간에 필요한 한 줄만 보여준다.

### Polish Guide
- 마지막 단계에서는 정렬, 간격, 라벨 길이, 표 헤더 높이, 버튼 높이, 포커스 링, 빈 상태 문구까지 전부 한 번 더 맞춘다.
- 픽셀 단위보다 역할 단위로 본다. 같은 역할이면 같은 크기와 간격이어야 한다.
- 모든 인터랙션에는 기본, hover, focus, active, disabled 상태가 있어야 한다.
- 실제 사용 기준으로 확인한다. 로그인, 검색, 표 스캔, 저장, 빈 상태, 오류 상태, 모바일 축소까지 직접 보고 어긋나는 디테일을 정리한다.

### Quieter Guide
- 조용한 콘솔이라는 기준은 유지하되, 힘이 빠진 화면이 되면 안 된다. 강한 지점은 더 분명하게, 나머지는 더 뒤로 보내는 방식으로 조절한다.
- 색이 너무 많거나 강조가 과하면 줄이되, 현재 위치·주 액션·위험 상태까지 같이 약하게 만들지 않는다.
- 과한 보더, 두꺼운 그림자, 넓은 여백, 무거운 헤더는 먼저 걷어낸다.
- quieter의 목표는 무채색화가 아니라 `정제된 위계`다.

### Overdrive Guide
- 이 제품에서 overdrive는 시각 효과가 아니라 `기술적으로 빠르고 정확하게 느껴지는 경험`이어야 한다.
- 허용되는 overdrive 방향은 대량 표 성능, 즉시 반응하는 필터, View Transition 기반의 짧은 패널 전환, 저장/선택 상태의 고급 피드백 정도다.
- shader, particle, cinematic scroll, audio 같은 연출형 overdrive는 이 제품 문맥에서는 금지한다.
- overdrive를 쓰려면 먼저 2~3개 방향을 비교하고, 성능·접근성·브라우저 호환성 기준으로 하나만 선택한다.

### Optimize Guide
- 첫 화면 로딩보다 먼저 체감되는 것은 표 스캔, 필터 반응, 메뉴 전환, 저장 피드백 속도다. 최적화 우선순위도 그 순서로 둔다.
- 레이아웃 계산을 반복하는 애니메이션, 과한 DOM 반복, 큰 표 전체 렌더링, 무거운 그림자/필터는 우선 제거 대상이다.
- 긴 표와 리스트는 필요하면 가상화까지 고려한다. 같은 카드 31개 반복처럼 DOM을 낭비하는 패턴은 금지한다.
- 이미지보다 텍스트 중심 제품이므로, JS/CSS 번들, 렌더링 비용, 입력 반응 속도(INP)를 먼저 본다.
- 목표는 “예쁘게 빠름”이 아니라 “업무가 끊기지 않게 빠름”이다.

### Frontend Quality Bar
- 누가 봐도 “AI가 만든 화면”처럼 보이면 실패다.
- 누가 봐도 “예전 사내툴”처럼 보이면 실패다.
- 첫 2초 안에 현재 위치, 핵심 작업, 다음 행동이 안 읽히면 실패다.
- 하루 종일 보는 화면으로 피곤하면 실패다.

## Design Context

이 제품은 접근성팀과 운영 담당자가 하루 종일 사용하는 `고대비·고밀도 운영 콘솔`이다.
빠르게 읽고 바로 상태를 즉시 판단하는 `상용 백오피스형 업무 도구`로 보여야 한다.

### Source Of Truth
이 프로젝트의 디자인 컨텍스트는 기존 내부 콘솔 감각이 아니라 KRDS 문서를 기준으로 다시 잡는다.

- 서비스 방향: `서비스 패턴`
- 화면 문법: `스타일 가이드`
- UI 구성: `컴포넌트 가이드`
- 구조 기준: `디자인 토큰`, `레이아웃`, `접근성 규칙`

### Product Type
`/my-works`는 일반 SaaS 랜딩이나 감성형 대시보드가 아니라, 로그인 후 반복 과업을 수행하는 `업무 서비스`다.
디자인의 목적은 보기 좋은 인상보다 `핵심 과업을 빨리 찾고, 이해하고, 수행하고, 종료하게 하는 것`이다.

### Users
- 주 사용자는 로그인 후 조회, 입력, 수정, 확인 업무를 수행하는 실무 사용자와 관리자다.
- 사용자는 시스템을 배우러 오는 것이 아니라 자기 일을 처리하러 들어온다.
- 초심자와 숙련자를 모두 견뎌야 하므로, 새로운 시각 실험보다 익숙한 구조와 일관성이 우선이다.
- 접근성은 선택이 아니라 기본이다. 저시력 사용자, 키보드 사용자, 색각 이상자를 기본 사용자군에 포함한다.

### Core Jobs
서비스의 핵심 흐름은 아래 과업 순서로 본다.

1. 접속한다.
2. 현재 위치와 상태를 파악한다.
3. 원하는 메뉴나 항목을 찾는다.
4. 데이터를 조회하고 비교한다.
5. 필요한 항목을 입력하거나 수정한다.
6. 결과와 상태를 확인하고 다음 행동으로 이동한다.

### Aha Moment
사용자가 느껴야 하는 핵심 가치는 `이 서비스는 복잡한 업무를 헤매지 않게 한다`이다.
첫 진입 후 몇 초 안에 현재 위치, 가능한 행동, 다음 단계가 읽혀야 한다.

### Experience Level
- 초심자는 설명이 없어도 길을 찾을 수 있어야 한다.
- 숙련자는 멈추지 않고 바로 작업할 수 있어야 한다.
- 따라서 문구보다 구조, 장식보다 위계, 새로움보다 예측 가능성을 우선한다.

### Brand Personality
`공공형`, `명료함`, `신뢰감`

- 친절함은 과잉 설명이 아니라 명확한 구조에서 나온다.
- 권위적이거나 구식인 인상은 피한다.
- 임시 사내 도구처럼 조악하거나 가벼운 인상도 금지한다.

### Style Direction
KRDS 기준에서 이 프로젝트는 독자 로고를 가진 운영 서비스/시스템에 가깝다.
따라서 `확장형 스타일`로 해석하되, 구조와 규칙은 KRDS를 따른다.

- 색상과 브랜드 표현은 확장할 수 있다.
- 그러나 구조, 위계, 접근성, 컴포넌트 역할, 토큰 체계는 임의로 바꾸지 않는다.

### Service Pattern Direction
`/my-works`는 화면 집합이 아니라 사용자 여정 기반 서비스로 설계한다.

- `방문`: 첫 화면에서 서비스 성격과 가능한 행동이 보여야 한다.
- `로그인`: 인증 화면도 서비스 흐름의 일부로 보여야 한다.
- `조회`: 검색, 표, 목록은 빠른 탐색과 비교를 돕는 구조가 우선이다.
- `입력/수정`: 폼은 긴 설명 없이 필요한 정보와 행동 순서가 자연스럽게 드러나야 한다.

### Layout Steering
레이아웃 작업 범위는 단일 파일이 아니라 서비스 전체 구조로 본다.

- 인증 후 공통 셸은 `SkipLink`, `Header`, `SideNavigation`, `Breadcrumb`, 사용자 메뉴, 전역 fetching 오버레이, `main`까지 포함한다.
- 비인증 화면도 별도 인증 전용 레이아웃 문법으로 함께 다룬다.
- `PageHeader`, `PageSection`, `PageFilterBar`, `PageResultBar`는 페이지 내부 기능이 아니라 공통 레이아웃 블록으로 취급한다.
- 조회형 기준 페이지의 상단 흐름은 `PageHeader -> PageSection(필터) -> PageResultBar -> 결과 영역` 순서로 읽혀야 한다.
- 레이아웃 판단 기준은 `src/styles.bak/original/`의 기존 밀도와 KRDS 문서를 함께 본다.
- 이 서비스에는 페이지 hero 개념을 두지 않는다. 인증 화면도 `상단 블록`, `제목 영역`, `인증 전용 레이아웃`으로만 부른다.
- 작업 중 유지가 필요한 디자인 판단은 이 문서에 기록한다.

### Layout Direction
레이아웃은 KRDS 서브 페이지 문법을 우선한다.

- `헤더 + 좌측 탐색 + 본문` 구조를 기본으로 한다.
- 본문은 12컬럼 기반으로 정렬한다.
- large 이상 화면에서 읽기 가능한 최대 폭과 최소 마진을 지킨다.
- 페이지 안에 별도의 hero 레이어를 만들지 않는다.
- 첫 화면은 소개가 아니라 실제 과업 요소가 차지해야 한다.

### Information Hierarchy
모든 화면은 아래 순서로 읽혀야 한다.

1. 현재 어디에 있는가
2. 여기서 무엇을 할 수 있는가
3. 지금 가장 중요한 정보는 무엇인가
4. 다음 행동은 무엇인가

제목 아래 긴 설명문, 분위기성 카피, 장식용 요약 카드로 위계를 흐리지 않는다.

### Visual System
- 기본 톤은 밝은 배경 기반의 공공 서비스 문법을 따른다.
- 강조는 `주 행동`, `현재 위치`, `선택 상태`, `상태 정보`에만 제한한다.
- 색은 의미 전달용으로만 사용한다.
- 그림자보다 표면, 경계, 간격으로 구조를 먼저 세운다.
- 엘리베이션은 메뉴, 드롭다운, 패널, 모달처럼 필요한 계층에만 제한적으로 사용한다.

### Typography
- 읽기 쉬운 본문체를 사용한다.
- 제목, 섹션 제목, 본문, 라벨, 메타의 역할을 분명히 분리한다.
- 작은 글씨로 밀어 넣는 방식은 금지한다.
- 장식적인 헤드라인보다 정보 해석 속도를 우선한다.

### Shape And Spacing
- 작은 반경과 안정적인 컨트롤 높이를 사용한다.
- 버튼과 입력은 같은 제품 문법으로 보여야 한다.
- 섹션 간격과 내부 간격은 반복 가능해야 한다.
- 카드 남발보다 섹션 구조와 여백, 경계로 질서를 만든다.

### Component Policy
- 헤더, 브레드크럼, 사이드 메뉴, 건너뛰기 링크는 서비스 탐색의 기본 골격이다.
- 표는 대량 정보 비교의 핵심 요소다.
- 배지와 상태 라벨은 보조 정보이며 제목보다 강하면 안 된다.
- 도움말은 요청형으로 제공한다.
- 과한 온보딩, 코치마크, 튜토리얼은 기본값이 아니다.

### Copy Direction
- 제목은 기능명과 과업명을 직접 말해야 한다.
- 버튼은 행동을 직접 표현해야 한다.
- 설명은 필요한 경우에만 짧게 쓴다.
- 사용자를 가르치려는 말투, 분위기성 문장, 중복 설명을 금지한다.
- 같은 개념은 같은 용어로만 쓴다.

### Accessibility Direction
- 접근성은 부가 옵션이 아니라 기본 전제다.
- 텍스트 대비, 포커스 표시, 키보드 탐색, 상태 전달, 논리 구조를 기본으로 맞춘다.
- 색만으로 상태를 전달하지 않는다.
- 선명한 화면 모드까지 수용 가능한 구조와 토큰 체계를 유지한다.

### Motion Direction
- 모션은 상태 변화 설명용으로만 사용한다.
- 장식용 등장 연출, 떠다니는 효과, 과한 스태거는 금지한다.
- 빠르고 짧게 끝나는 피드백만 허용한다.
- `prefers-reduced-motion`를 존중한다.

### Token Direction
토큰은 `primitive -> semantic -> component` 계층으로 관리한다.

- 색상, 타이포, 간격, 형태는 원시 토큰으로 정의한다.
- 역할 기반 의미 토큰을 별도로 둔다.
- 버튼, 입력, 표, 패널, 네비게이션 토큰은 의미 토큰 위에서만 만든다.
- 즉흥적인 이름과 화면별 임시값 누적을 금지한다.

### Anti-Patterns
- SaaS형 hero
- 동일 카드 반복 대시보드
- 박스 안의 박스 안의 박스
- 장식용 그라데이션과 유리질감
- 과한 그림자와 떠 있는 위젯 인상
- 제목 아래 장황한 소개 문단
- 페이지마다 다른 버튼/필터/패널 문법
- 색으로만 의미를 전달하는 상태 표현
- 컴포넌트 토큰 없이 화면별 임시 스타일 누적

### Design Principles
1. 화면은 과업 중심으로 조직한다.
2. 사용자가 길을 잃지 않도록 탐색 구조를 먼저 세운다.
3. 구조와 컴포넌트는 KRDS 역할 문법을 따른다.
4. 설명보다 위계와 배치로 이해시킨다.
5. 접근성과 일관성을 미학보다 우선한다.
6. 토큰과 컴포넌트 단위로 재사용 가능하게 만든다.
7. 강한 강조는 소수의 핵심 요소에만 준다.
8. 서비스는 친절해야 하지만 가볍거나 장식적이면 안 된다.

## Implementation Guardrails

### Class Naming Scope
- 사용자가 `클래스명`을 이야기하면 작업 범위는 클래스명과 그에 연결된 셀렉터 이름으로만 한정한다.
- 디자인 토큰(`var(--...)`), 색상값, 간격값, 타이포 값, 토큰 파일 구조는 클래스명 작업 범위에 포함하지 않는다.
- BEM 적용 시 기준은 `block`, `block__element`, `block--modifier`이며, modifier는 base 클래스와 함께 사용한다.
- 클래스명 정리 중에도 스타일 값 자체를 임의로 재설계하거나 토큰 체계를 바꾸지 않는다.

### Inline Style And CSS-in-JS Rule
- 인라인 스타일과 CSS-in-JS는 기본적으로 금지한다.
- 다만 JS 제어를 위해 구조적으로 불가피한 경우는 예외로 허용한다.
- 예외가 아니라면 `style={...}`, `<style>`, CSS-in-JS 패턴은 일반 CSS/SCSS 클래스로 옮긴다.

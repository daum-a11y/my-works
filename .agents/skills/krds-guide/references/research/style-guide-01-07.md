# KRDS 스타일 가이드 01~07 정리

## 1. 범위와 해석 기준

검토 범위:

- `style_01`: 디자인 스타일 소개
- `style_02`: 색상
- `style_03`: 타이포그래피
- `style_04`: 형태
- `style_05`: 레이아웃
- `style_06`: 아이콘
- `style_07`: 디자인 토큰

보조 메모:

- `style_01` 메뉴에는 `엘리베이션`과 `선명한 화면 모드`가 핵심 항목으로 잡혀 있으나,
  현재 상세 페이지 분리는 `style_08` 엘리베이션, `style_07` 디자인 토큰 내 모드 전환 문단으로 흡수되어 있습니다.
- 따라서 01~07만 기준으로 스킬을 만들 경우, 엘리베이션은 누락될 수 있으므로 별도 경고를 넣는 편이 안전합니다.

## 2. 공통 프레임

출처: <https://www.krds.go.kr/html/site/style/style_01.html>

KRDS 스타일 가이드는 다음 세 가지 목적을 반복해서 강조합니다.

- 일관성: 기관이 달라도 정부 서비스라는 시각적 신뢰를 유지
- 접근성: 전자정부 품질지침과 WCAG AA 기준을 실무적으로 반영
- 효율성: 디자이너와 개발자가 같은 기준으로 빠르게 구현

또한 기관 적용 방식은 두 축으로 분리됩니다.

### 표준형 스타일

- 대상:
  정부 상징 로고를 쓰는 중앙행정기관, 특별지방행정기관, 부속기관 등의 대표/운영 웹사이트, 모바일 웹/앱
- 기준:
  색상, 타이포그래피, 형태, 레이아웃, 아이콘, 엘리베이션 등 스타일 요소와 규칙을 필수 준수

### 확장형 스타일

- 대상:
  독자 로고를 쓰는 중앙기관 운영 서비스/시스템, 공공기관 대표/운영 서비스/시스템 등
- 기준:
  기본 규칙은 지키되 기관의 정체성을 반영해 확장 가능

스킬 관점 해석:

- 결과물을 만들 때 먼저 `표준형`인지 `확장형`인지 판단해야 합니다.
- 판단 불가 시 기본값은 `표준형`으로 두는 것이 보수적입니다.
- 확장형이어도 접근성과 구조 규칙은 약화하면 안 됩니다.

## 3. style_01: 디자인 스타일 소개

출처: <https://www.krds.go.kr/html/site/style/style_01.html>

핵심 메시지:

- 스타일은 디자인 시스템의 핵심 요소
- 기관 커스터마이징 허용 범위와 절대 준수 기준을 분리
- 실제 작업 전에 자체 검증 체크리스트 사용 권장

소개 페이지가 안내하는 스타일 구성:

- 색상
- 타이포그래피
- 형태
- 레이아웃
- 아이콘
- 엘리베이션
- 디자인 토큰

소개 페이지가 직접 강조하는 검증 항목:

- 글자 크기
- 색상 대비
- 아이콘 크기
- 줄 높이

스킬 반영 포인트:

- 출력 전에 반드시 접근성/일관성 검증 절차를 넣어야 합니다.
- 스타일 판단은 미감보다 `기준 충족 여부`를 먼저 보게 만들어야 합니다.

## 4. style_02: 색상

출처: <https://www.krds.go.kr/html/site/style/style_02.html>

### 4.1 팔레트 구조

- 표준형 팔레트는 명도 단계를 기반으로 구성
- Gray는 `0(white)`와 `100(black)`을 포함
- Gray를 제외한 다른 계열은 11단계 중심으로 구성

핵심은 단순 색상표가 아니라 `대비를 계산 가능한 체계`라는 점입니다.

### 4.2 매직 넘버

KRDS 색상 문서에서 가장 실무적인 개념입니다.

- 매직넘버 40 = 대비 3:1
- 매직넘버 50 = 대비 4.5:1
- 매직넘버 70 = 대비 7:1
- 매직넘버 90 = 대비 15:1

스킬에 반드시 들어가야 할 규칙:

- 텍스트/아이콘/보더/상태색은 단순 취향이 아니라 이 대비 규칙 기준으로 판단
- 색상 선택 시 “왜 이 값인가”를 매직 넘버 기준으로 설명 가능해야 함

### 4.3 접근성 주의 사항

- 단계별 색상은 명도 대비를 먼저 보고 설계
- 색맹 시뮬레이터 사용 권장
- 기본 모드와 선명한 화면 모드에서 배경과 유사한 `5`, `95` 단계 사용 주의

### 4.4 색약 사용자 고려

문서가 직접 강조하는 부분:

- 적록 계열 구분 문제를 우선 고려
- 색상만으로 상태를 전달하지 말고 텍스트/아이콘으로 보완
- 적색/녹색 조합은 특히 보수적으로 사용

스킬 규칙으로 바꾸면:

- 상태 전달은 `색상 + 텍스트 + 아이콘`
- 성공/위험/경고 구분을 색만으로 맡기지 않음

### 4.5 색상 시스템

문서가 정의하는 역할 기반 그룹:

- Gray: 배경, 텍스트, 구분선, 중립 정보
- Primary: 주요 상호작용
- Secondary: 보조 상호작용 및 기본 UI
- Accent: 강조 요소, 제한적 사용
- Graphic: 그래프/배너/일러스트 등 보조 시각요소
- System: 상태 전달용 (`Danger`, `Warning`, `Success`, `Info`)

### 4.6 주요 색상 운용

- Primary:
  가장 주목받아야 하는 액션/중요 정보 강조
- Secondary:
  보조 역할, 배경/구분/덜 중요한 액션
- Gray:
  콘텐츠 집중을 방해하지 않는 중립색

색상 비율 가이드:

- 60%: 배경 및 중립색
- 30%: 보조 색상
- 10%: 주요 상호작용 색상

이 비율은 스킬의 “과도한 포인트 컬러 남용 방지” 규칙으로 쓰기 좋습니다.

### 4.7 시스템 색상

예시로 `Danger`가 구체적으로 설명됩니다.

기본 모드 예:

- 아이콘: `danger-50`
- 텍스트: `danger-60`
- 배경: `danger-5`
- 보더: `danger-10`

선명한 화면 모드 예:

- 아이콘: `danger-20`
- 텍스트: `danger-20`
- 배경: `danger-95`
- 보더: `danger-90`

스킬 반영 포인트:

- 상태 박스/배너를 만들 때 텍스트, 배경, 보더, 아이콘을 묶어서 판단
- 한 값만 바꾸는 식으로 설계하지 않음

### 4.8 글자/보더/아이콘 대비 규칙

문서에 명시된 기준:

- 글자 색상:
  - 기본 모드 최소 매직넘버 40
  - 본문은 매직넘버 50
  - 선명한 화면 모드는 매직넘버 70
  - 본문은 90(15:1)
- 인풋 보더:
  - 기본 모드 50
  - 선명한 화면 모드 50 + 두께 변화
- 아이콘:
  - 기본 모드 40
  - 선명한 화면 모드 70

### 4.9 사용 가이드 요약

- 시스템 색상은 시각 보조수단과 함께 사용
- 주요 색상은 진짜 중요한 액션에만 사용
- 대비율은 문서 수치를 그대로 준수
- 새 색상 추가는 재사용성이 높을 때만 허용

## 5. style_03: 타이포그래피

출처: <https://www.krds.go.kr/html/site/style/style_03.html>

### 5.1 기본 서체

- 표준형:
  국문/영문 모두 `Pretendard GOV`
- 확장형:
  가독성 높은 고딕 계열 권장

문서가 추천하는 고딕 계열 예:

- Noto Sans
- 나눔고딕
- 스포카 한 산스

### 5.2 글자 두께

표준형 기본:

- Regular 400
- Bold 700

실무 메모:

- 너무 얇은 글자(thin/light) 지양
- 너무 무거운 extra bold도 지양
- 텍스트 두께 종류는 최대 4종 정도로 제한하는 것이 이상적
- 문서가 이상적 예시로 `regular(400), medium(500), bold(700)`을 제시

### 5.3 줄 간격

- 최소 `150%` 이상
- `px`보다 `em`, `%` 같은 상대 단위 권장
- 시각장애/난독증 사용성에 직접 영향

Figma 설정 메모:

- Vertical Trim은 `Standard` 권장
- 개발 `line-height`와의 불일치 방지 목적

### 5.4 기본 폰트 크기

- 본문 최소 `16px` 이상
- `Pretendard GOV`는 상대적으로 작게 느껴져 기본값을 `17px`로 사용

스킬 반영 포인트:

- KRDS 스타일 결과물에서 본문을 14px, 15px로 시작하는 패턴은 기본적으로 경고 대상
- 보조 정보가 아닌 본문은 17px 기준으로 보는 편이 안전

### 5.5 단위

문서 권장:

- 구현 시 `rem` 사용
- 루트값 `62.5% = 10px` 기준 사용 가능

표시 예:

- 100% = 16px
- 62.5% = 10px
- 50% = 8px

### 5.6 글자 스케일

구조:

- Display
- Heading
- Body

비율 원칙:

- Heading과 Body 간 차이는 `1.25 ~ 1.5배` 권장

#### Display

- large: 60 / 44
- medium: 44 / 32
- small: 36 / 28

형식: `pc / mobile`

#### Heading

- xlarge: 40 / 28
- large: 32 / 24
- medium: 24 / 22
- small: 19 / 19
- xsmall: 17 / 17
- xxsmall: 15 / 15

Heading 사용 구조:

- h1: xlarge-large
- h2: large-medium
- h3: medium-small
- h4: small-xsmall
- h5: xsmall-xxsmall

#### Body

- large-bold: 19
- medium-bold: 17
- small-bold: 15
- xsmall-bold: 13
- large: 19
- medium: 17
- small: 15
- xsmall: 13

Body 사용 가이드:

- large: 요약/중요 정보
- medium: 표준 본문
- small: 덜 중요한 설명
- xsmall: 주석/보조 설명

### 5.7 글자 색상과 대비

문서 기준:

- AA:
  - 큰 텍스트 3:1
  - 일반 텍스트 4.5:1
- AAA:
  - 큰 텍스트 4.5:1
  - 일반 텍스트 7:1

큰 텍스트 기준:

- 18pt 이상
- 또는 bold 기준 14pt 이상
- 픽셀 환산:
  - 18pt = 24px
  - 14pt = 18.66px

선명한 화면 모드 기준:

- 본문 텍스트: 15:1
- 헤딩/레이블/아이콘: 7:1
- 시각적 보조 수단: 4.5:1

## 6. style_04: 형태

출처: <https://www.krds.go.kr/html/site/style/style_04.html>

### 6.1 형태가 전달하는 인상

문서는 형태를 정서/브랜드 인상과 연결합니다.

- 각진 형태:
  전문성, 정확성, 효율성
- 둥근 형태:
  친근함, 안정감, 부드러움

스킬에서 필요한 판단:

- 정부 대표 서비스처럼 강한 공신력이 필요한 화면은 과도하게 둥글게 만들지 않음
- 친화적 브랜드여도 KRDS 표준형 상한을 넘지 않음

### 6.2 표준형 래디어스 체계

레벨 5단계:

- xsmall
- small
- medium
- large
- xlarge

값 범위:

- `2px ~ 12px`
- 최댓값 `12px`

문서가 제시하는 대표 적용:

- Xsmall 2px:
  인디케이터, 배지, 프로그레스바
- Small 4px:
  태그, 칩, 체크박스, 라디오버튼, 스위치
- Medium 6px:
  버튼, 텍스트 입력, 텍스트 영역, 셀렉트, 스텝 인디케이터, 페이지네이션
- Large 10px:
  카드, 다이얼로그
- Xlarge 12px:
  배너, 다이얼로그, 바텀 시트

### 6.3 확장형 래디어스

확장형은 커스터마이즈 허용하지만 규칙이 있습니다.

- 기준 컴포넌트의 높이 대비 비율로 계산
- 짝수값 사용 권장
- 홀수/소수점이 나오면 근사치로 올림 후 짝수값 사용

### 6.4 px vs %

- px:
  버튼, 입력 필드 같이 일관성이 중요한 컴포넌트에 적합
- %:
  완전한 원형이 필요한 경우에만 제한적으로 사용

문서가 경고하는 % 사용 리스크:

- 컴포넌트별 비율 변동
- 텍스트 일관성 부족
- 작은 컴포넌트에서 과도하게 둥글어짐

### 6.5 사용 가이드

- 함께 쓰이는 비슷한 크기의 요소는 같은 래디어스 사용
- 큰 컨테이너는 비율에 맞춰 래디어스도 커짐
- 내부 콘텐츠를 침범할 정도의 과한 래디어스 금지
- 둥근 형태를 쓸수록 내부 패딩을 충분히 확보

## 7. style_05: 레이아웃

출처: <https://www.krds.go.kr/html/site/style/style_05.html>

### 7.1 그리드 요소

핵심 구성:

- Screen margin
- Column
- Gutter

### 7.2 스크린 마진

- large: 최소 24px
- medium: 최소 24px
- small: 최소 16px

의미:

- 화면 가장자리와 콘텐츠 충돌 방지
- 모바일 터치 오류 방지

### 7.3 칼럼

- large: 12~16
- medium: 8~12
- small: 4~6

중요 규칙:

- 동일 화면에서 칼럼 수 혼용 금지
- 고정 픽셀이 아니라 백분율 기반으로 유동 배치

### 7.4 가터

- large: 최소 16px / 최적 24px
- medium: 최소 16px / 최적 24px
- small: 최소 16px / 최적 16px

규칙:

- 칼럼 너비의 절반 이하
- 너무 넓거나 좁지 않게 유지

### 7.5 브레이크포인트

일반 브레이크포인트 표:

- xsmall: `-359px`
- small: `360px-`
- medium: `768px-`
- large: `1024px-`
- xlarge: `1280px-`
- xxlarge: `1440px-`

표준형 스타일 브레이크포인트:

- small: 360px-, 4 columns, gutter 16, margin 16
- medium: 768px-, 8 columns, gutter 16, margin 24
- large: 1024px-, 12 columns, gutter 24, margin 24
- xlarge: 1280px-, 12 columns, gutter 24, margin 24

추가 메모:

- 표준형은 `xsmall`을 최적화 대상에 포함하지 않음
- 확장형은 3~6단계 브레이크포인트를 자유롭게 운용 가능

### 7.6 콘텐츠 영역

문서는 콘텐츠 영역을 두 가지로 구분합니다.

- 제한된 콘텐츠 영역(고정형)
- 풀 반응형/유연한 콘텐츠 영역(가변형)

표준형 기본:

- 최대 콘텐츠 너비 `1200px`
- `24px` 스크린 마진 고려
- `24 + 1200 = 1248px` 이상에서 고정 유지

### 7.7 서브 페이지 레이아웃

적용 조건:

- large 이상, 즉 `1024px+`

구성 가능 요소:

- Header
- Left menu
- Main contents
- Right menu
- Footer

운용 메모:

- 왼쪽 메뉴:
  깊은 IA에 적합
- 오른쪽 메뉴:
  콘텐츠 내 탐색, 도움 패널, 빠른 메뉴
- 양쪽 메뉴 동시 사용:
  작은 화면에서 가독성 저하 주의

### 7.8 간격

핵심 원칙:

- `8-point grid system`
- gap, padding, responsive spacing 포함
- primitive token `number`를 사용

반응형 간격 범위:

- Mobile small: 360px 이상
- Mobile medium: 768px 이상
- PC large: 1024px 이상, 콘텐츠 기준 1200px
- PC xlarge: 1280px 이상

문서가 구체적으로 적어둔 예:

- 카드 리스트 gap: `gap-7 = 24px`
- 인풋 가로 조합: `gap-5 = 16px`
- 인풋 세로 조합: `gap-5 ~ gap-7 = 16px ~ 24px`
- 체크박스/라디오 리스트: `gap-5 ~ gap-7`
- 체크박스/라디오 가로형: `gap-7` 이상

## 8. style_06: 아이콘

출처: <https://www.krds.go.kr/html/site/style/style_06.html>

### 8.1 기본 철학

- 기관별 일반 아이콘은 달라질 수 있음
- 시스템 아이콘은 공통 언어로 통일
- 우선순위는 일관성, 접근성, 가독성

### 8.2 파일 형식

- SVG 사용

이유:

- 확대/축소 시 화질 유지
- 파일 경량
- CSS 스타일링 가능

### 8.3 크기

기준 크기:

- 24px

주요 크기:

- 16px
- 20px
- 24px
- 32px
- 40px

규칙:

- 4px 또는 8px 배수 사용

### 8.4 키라인

아이콘은 다음 구조를 가집니다.

- Live Area
- Padding

키라인 역할:

- 서로 다른 모양의 아이콘이 같은 크기에서 같은 시각 면적으로 보이도록 조정
- 필요 시 일부 요소 길이를 미세 조정 가능

### 8.5 두께

권장 범위:

- 24px 아이콘 기준 `1.5px ~ 2px`

표준형 기준:

- 24px 기준 `1.6px`

### 8.6 둥글기와 join/end cap

- 24px 아이콘에서 `radius 5px` 이상은 형태 무리 여부 점검
- 표준형은 `end cap`을 둥근 형태로 사용

join 관련 메모:

- `join: miter`는 대략 radius 0.8과 유사
- `join: round`는 radius 0과 유사한 해석 예시가 문서에 제시됨

### 8.7 디테일 규칙

- off slash:
  좌상단 → 우하단 방향
- off slash 여백:
  선 두께의 1/2, 즉 `0.8px`
- stack:
  좌 → 우 순서 통일
- stack 겹침 여백:
  `1.6px`

### 8.8 아이콘 색상 대비

- 기본 모드: 인접 색상과 최소 `3:1`
- 선명한 화면 모드: `7:1`

또한 문서는 다음을 요구합니다.

- 같은 크기 아이콘은 같은 두께만 사용
- Perspective view 금지, Front view 사용
- Line type / Fill type 구분
- 텍스트와 함께 쓰면 중앙 정렬
- 텍스트와 함께 쓸 때는 매칭된 사이즈만 사용

## 9. style_07: 디자인 토큰

출처: <https://www.krds.go.kr/html/site/style/style_07.html>

### 9.1 정의

디자인 토큰은 색상, 글자, 간격 같은 스타일 속성을 변수화한 코드입니다.

문서 표현을 실무적으로 바꾸면:

- 디자인 값을 직접 박지 않고
- 이름 있는 변수로 관리하고
- 디자인/개발 도구에서 공통 언어로 사용한다는 뜻입니다.

### 9.2 토큰을 쓰는 이유

- 일관성 유지
- 협업 강화
- 접근성 및 표준화 지원

### 9.3 KRDS 토큰 흐름

문서 기준 흐름:

1. 디자인 툴 토큰
2. JSON 추출
3. CSS 변수 변환
4. HTML 반영

### 9.4 토큰 레벨

#### Primitive token

- 기본 속성 레벨
- 예:
  `primary-50`, `gray-5`, `number-4`

포함 예:

- color
- typo
- space
- radius

중요:

- 직접 쓰는 값이 아니라 다른 토큰의 기반값

#### Semantic token

- 특정 맥락의 의미를 가진 토큰
- 예:
  `color-icon-primary`, `color-border-gray-light`

포함 예:

- 배경 색상
- 아이콘 색상
- 카드 패딩
- 작은 컴포넌트 래디어스

#### Component token

- 버튼, 인풋 등 개별 컴포넌트의 세부 속성
- 코드에서 작성/관리

### 9.5 표기 규칙

문서 핵심:

- 큰 개념에서 세부 요소로 좁혀가는 구조
- 시멘틱 토큰과 컴포넌트 토큰 표기 분리

컴포넌트 토큰 규칙:

- 컴포넌트명을 먼저 적음
- 컴포넌트명 뒤에 대시 2개 사용
- 새 속성은 가능하면 CSS 속성명을 차용
- 좌우 동일 값은 `-x`
- 상하 동일 값은 `-y`

예:

- `--namespace-component--type-padding-x`

### 9.6 구현 방식

- CSS variable로 직접 정의 가능
- 반복 규칙이 많으면 SCSS 배열 + `@each` 사용 가능
- 컴포넌트 토큰은 각 컴포넌트 클래스 내부에서 독립 관리

문서가 따로 명시한 예외:

- 헤더/푸터 같은 레이아웃형 컴포넌트
- 외부 플러그인을 쓰는 캐러셀

이 경우 컴포넌트 토큰보다 시멘틱 토큰 중심 반영

### 9.7 모드 전환

`style_07` 안에 사실상 선명한 화면 모드와 반응형 모드 개념이 같이 들어 있습니다.

#### 선명한 화면 모드

- 기본 모드와 선명한 화면 모드 두 가지 제공
- 보더 값, 색상 값이 모드별로 전환
- 고대비 명도 기준 준수 필요

#### 반응형 모드

- large / small 디바이스 대응
- 글자 크기
- 간격
- 카드 패딩

즉, 스킬이 토큰을 다룬다면 단순 색 변수만이 아니라 `mode`와 `responsive`도 같이 다뤄야 합니다.

## 10. 스킬 작성용 핵심 체크리스트

### 반드시 규칙으로 넣을 것

- 본문 글자 최소 16px, KRDS 표준은 17px 기준
- line-height 최소 150%
- Primary/Secondary/Gray 역할 분리
- 상태 표현은 색상 단독 사용 금지
- 래디어스 표준형 상한 12px
- 8-point spacing 사용
- 표준형 레이아웃 최대 1200px 콘텐츠 영역
- 아이콘 기준 크기 24px, 두께 1.6px
- 디자인 토큰은 primitive → semantic → component 3단계 구조 유지

### 판단 분기에서 먼저 물어야 할 것

1. 표준형인가 확장형인가
2. 화면이 웹인가 모바일인가
3. 기본 모드만 필요한가, 선명한 화면 모드도 필요한가
4. 결과물이 토큰까지 포함한 구현인가, 시안/가이드 수준인가

### 스킬이 경고해야 할 것

- 과도한 포인트 컬러 사용
- 4px 이하 또는 12px 초과 래디어스 남용
- 본문 15px 이하
- 대비 기준 미달
- 아이콘 line/fill 혼용 혼란
- xsmall 디바이스까지 표준형으로 무리하게 동일 레이아웃 유지
